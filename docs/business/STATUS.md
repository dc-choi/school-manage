### 신당동/금호동 — 계정 모델 문제 발견 (02-20)
- **문제**: 한 단체에서 여러 명이 각각 개별 가입
- "공유 계정" 모델이 사용자 행동과 맞지 않음 → 각자 개별 가입하는 게 자연스러운 행동

### 계정 모델 전환 (2026-02-20)

#### 문제
1. 한 단체에서 여러 명이 각각 가입 — "공유 계정" 모델이 사용자 행동과 맞지 않음
2. 그룹/멤버 개념을 이해 못함 — 제품의 정보 구조가 사용자 멘탈 모델과 불일치
3. 학년(중1반, 중2반) 외에 부서(전례부, 봉사부, 홍보부) 그룹핑이 존재 — 한 학생이 여러 그룹에 동시 소속 (현재 1:N → N:M 필요)

#### 전환 방향
- 기존: Account (공유 계정) → Group (1:N) → Student → Attendance
- 전환: Account (개인) → Organization (본당/모임) → Group (N:M) ↔ Student → Attendance
- 가입 후 본당/모임을 찾아 합류하거나 새로 생성 (카카오톡 그룹채팅 패턴)
- Group은 학년(중1반, 중2반)과 부서(전례부, 봉사부) 두 축을 모두 지원
- 한 학생이 여러 그룹에 소속 가능 (중2반 + 전례부)
- 출석은 그룹별/전체 미사 모두 지원 가능해야 함

#### 해결되는 것
| 기존 문제 | 해결 방식 |
|-----------|-----------|
| 한 단체에서 여러 계정 생성 | 각자 가입 후 같은 본당에 합류 → 정상 동작 |
| 그룹/멤버 개념 혼란 | "우리 본당 찾아서 들어가기" → 자연스러운 흐름 |
| 학년만 지원, 부서 그룹핑 불가 | 학년/부서 두 축 지원 + N:M 관계로 다중 소속 가능 |
| 교리교사 교체 시 인수인계 (리스크 7) | 개인 계정 유지, 역할만 이전 |
| WAU 측정 부정확 | 개인별 로그인 = 실제 활성 사용자 |
| 데이터 소유권 불명확 | 본당 단위 소유, 개인은 접근 권한 |

#### 새로 발생하는 리스크
- 권한 관리 복잡성 (관리자/일반 교사 역할 구분)
- 기존 데이터 마이그레이션 (장위동 등)
- 온보딩 단계 추가 (가입 → 본당 찾기/생성 → 합류)

#### 사업 모델 영향
- 결제 주체 재정의 필요 (본당 단위 구독 vs 개인 프리미엄)
- 초대 기반 바이럴 루프 가능 (한 사람 가입 → 같은 본당 교리교사 초대)
- GTM 확산 구조 개선
