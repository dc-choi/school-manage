# 브레인스토밍 결과 보고서

## 주제
행사 관리 기능을 통한 수익화 — 인구 감소에 따른 공소 활용도 하락, 교리교사 행사 준비 부담, 중고등부 피정 감소 상황에서 행사 관리가 가장 수익화 명분이 있는 기능인가?

## 토론 요약

### Round 1
- 기획자가 3개 아이디어 제안: 피정 모집 보드, 공소 행사 연결 플랫폼, 행사 준비 체크리스트
- 비판자 핵심 지적: 세 아이디어 모두 계정 모델 전환/WAU 10명 달성이라는 현재 최우선 과제보다 먼저 논의됨. 피정 보드는 공급자 온보딩 문제, 공소 연결은 양면 시장 콜드 스타트, 체크리스트는 실행 순서 문제
- 합의: 공소 연결(아이디어 2) 원안 기각, 체크리스트는 계정 모델 전환 후 실행, WAU 10명이 선행 조건

### Round 2
- 기획자 개선: 피정 보드 → 크롤러 기반 카드 축소, 공소 연결 → 장소 메모 필드로 축소, 체크리스트 → 실행 시점 재정의
- 비판자 핵심 지적: 크롤링 법적/기술적 리스크, Event 엔티티 신설이 세 아이디어 공통 전제, "간단한 행사 메모 카드" 하나로 통합 제안
- 합의: 크롤링 기각 → 관리자 직접 입력, 세 아이디어 통합 방향

### Round 3
- 기획자: 통합안 제시 — 행사명/날짜/장소/메모 + 행사 출석 기록 + 통계 분리. 설계 4원칙(크롤링 없음, Event 1개, 병렬 진행, 체크리스트 제외)
- 비판자 최종 지적: Event를 Group 하위에 붙이면 계정 전환 시 마이그레이션 2회. "좋은 기능"과 "지금 해야 하는 기능" 구분 필요. 조건부 승인 제시
- 기획자 수용: 지금 착수 안 함. Event는 Organization 하위. 착수 조건 2가지 확정
- 최종 합의 도달

## 아이디어 목록

### [아이디어 1] 피정 모집 보드 (원안)
- 한 줄 요약: 교구/지구 피정 일정을 한곳에 모아 교리교사가 조회·신청하는 게시판
- 사용자 가치: 피정 정보 분산 해소
- 실현 방법: UGC 게시판 + 필터링
- 차별점: 피정 정보 → 학생 선택 → 이력 기록 연결
- 판정: **기각 → 행사 메모 카드로 흡수**. 공급자 온보딩 문제, 크롤링 리스크, 파일럿 데이터 근거 없음

### [아이디어 2] 공소 행사 연결 플랫폼 (원안)
- 한 줄 요약: 지방 공소와 서울 본당 중고등부를 피정/캠프 장소로 매칭
- 사용자 가치: 장소 조달 어려움 + 공소 활용도 감소 동시 해결
- 실현 방법: 시설 등록 → 검색 → 문의 → 행사 등록
- 차별점: 가톨릭 구조 특화, 독보적
- 판정: **기각**. 양면 시장 콜드 스타트, 교구 승인 구조, 타겟 사용자 변경, 3개월 내 불가. 3단계 이후 재검토 항목으로 분류

### [아이디어 3 → 통합] 행사 메모 카드
- 한 줄 요약: 행사 날 출석을 평소 출석과 분리 기록하고, 행사 정보를 한 화면에서 조회
- 사용자 가치: 행사 출석 분리 기록, 사제 보고용 행사 통계 자동 집계, 행사 메모 영구 보존
- 실현 방법: Event 엔티티 1개(행사명/날짜/장소/메모) + Attendance.event_id 외래키 + 동일 체크 UI + 통계 분리
- 차별점: 카카오톡/엑셀은 행사 참여자와 출석 데이터 연결 불가. 이 기능은 행사-출석이 같은 맥락
- 판정: **조건부 승인**

## 착수 조건 (행사 메모 카드)

| 조건 | 기준 | 현재 상태 |
|------|------|----------|
| 계정 모델 전환 SDD 완료 | Organization 엔티티 확정, Group N:M 구조 확정 | 미착수 |
| 수요 검증 | 장위동/정릉4동 교리교사 직접 질문, 긍정 응답 2곳 이상 | 미실시 |

## 설계 원칙 (착수 시 적용)

1. 크롤링 없음. 관리자 직접 입력. 필드: 행사명(필수), 날짜(필수), 장소/메모(선택)
2. Event 엔티티 1개 + Attendance.event_id 외래키. 스키마 변경 최소화
3. Event는 Organization 하위로 설계 (Group 하위 기각)
4. 체크리스트 MVP 제외. 행사 참여 출석 기록 + 행사 메모만 범위
5. 동일 체크 UI 재사용 (학습 비용 0)

- 긍정 응답 → 착수 (계정 모델 전환 SDD에 Event 설계 포함)
