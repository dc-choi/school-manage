# 브레인스토밍 공통 규칙

## 프로젝트 컨텍스트

이 프로젝트는 **가톨릭 모임 운영 도구**(출석부 프로그램)다.
토론 시 반드시 아래 사업 문서를 읽고 현재 상황을 파악한 뒤 발언해라:

| 문서                                             | 용도                              |
| ------------------------------------------------ | --------------------------------- |
| `docs/business/STATUS.md`                        | 현재 목표, 파일럿 현황, 오픈 이슈 |
| `docs/business/README.md`                        | 비전, 핵심 문제, 문서 인덱스      |
| `docs/business/1_market/market.md`               | 시장/경쟁 분석                    |
| `docs/business/2_bm/bm.md`                       | 가치 제안, 수익 모델              |
| `docs/business/4_risk/risks.md`                  | 리스크/가정                       |
| `docs/business/1_market/competitive-analysis.md` | 경쟁사 상세 분석                  |

## 토론 규칙

1. 모든 발언은 근거를 포함해라. "좋다/나쁘다"만 쓰지 마라
2. 상대 의견을 먼저 요약한 뒤 자신의 의견을 말해라
3. 동의할 점은 명확히 인정해라
4. 반박 시 대안을 함께 제시해라
5. 감정적 표현을 배제하고 논리적으로 서술해라
6. **1인 개발 프로젝트**임을 항상 고려해라 (리소스 제약)
7. 모든 문서는 **한글로 작성**해라

## 할루시네이션 가드 (HIGH 우선)

기획자/비판자 모두 _주장형_ 발화를 만들어야 한다는 압박을 받는다. 그래서 false positive(없는 데이터로 우기기, 추측한 리스크 단정) 위험이 크다. 다음을 반드시 지킨다 — 위반 발언은 라운드 요약에서 폐기 대상.

1. **근거 형식 강제.** "근거 포함"만으론 부족. 셋 중 하나 형태로 인용:
    - 사업 문서 인용: `STATUS.md L42` 또는 `>` 인용구
    - 파일럿 데이터: 본당명/규모/기간 명시 (예: "정자동 60명, 2개월 운영")
    - 외부 레퍼런스: 출처 + 링크 (없으면 사용 X)
2. **"가능성 있음"·"리스크일 수 있음" 단독 발언 금지.** 추정 리스크는 *구체 시나리오 + 발생 조건*까지 명시. 둘 다 못 대면 발언 X.
3. **사업 문서에 없는 수치/타겟/경쟁사 만들지 마라.** 토론 전 위 사업 문서 목록을 _실제로 읽고_ 발언. 안 읽었으면 "확인 필요"로 보류.
4. **자신 없으면 "근거 부족 — 검증 필요"라고 말해라.** 억지로 ✅/❌ 판정 채우지 마라. false 데이터는 다음 라운드를 오염시킨다.
5. **비판자는 *없는 문제*를 만들지 마라.** 비판은 기획자 발언의 *실제 약점*에만. "이런 가정도 있을 수 있다"는 비판 X.

## 아이디어 평가 축

모든 아이디어는 다음 4가지 축으로 평가한다:

| 축            | 질문                                                 |
| ------------- | ---------------------------------------------------- |
| 사용자 가치   | 교리교사/모임 운영자의 실제 페인포인트를 해결하는가? |
| 기술 실현성   | 1인 개발로 3개월 내 MVP를 만들 수 있는가?            |
| 시장 차별성   | 가톨릭 전용 도구로서 기존 범용 앱과 무엇이 다른가?   |
| 비즈니스 모델 | 수익 모델 또는 성장에 어떻게 기여하는가?             |

## 라운드 출력 형식

각 라운드는 다음 구조로 기록해라:

```
## Round N

### 기획자 발언
(planner의 아이디어 또는 반론)

### 비판자 발언
(critic의 비판 및 질문)

### 라운드 요약
- 합의된 사항: ...
- 미합의 사항: ...
- 다음 라운드에서 다룰 것: ...
```

## 파일 저장 규칙

- 디렉토리: `docs/brainstorm/YYYY-MM-DD/` (날짜별 폴더)
- 파일명: `<주제-slug>.md` (예: `revenue-model.md`, `acquisition-retention.md`)
- 같은 날 여러 주제를 다룬 경우 같은 폴더에 주제별 파일 생성

```
docs/brainstorm/
├── 2026-02-23/
│   └── event-management.md
├── 2026-03-16/
│   ├── acquisition-retention.md
│   ├── catholic-news-tracker.md
│   └── revenue-model.md
```

```

(브레인스토밍 주제)

(3라운드 핵심 내용 요약)

## 아이디어 목록

- 한 줄 요약
- 사용자 가치
- 실현 방법
- 차별점

(아이디어별 반복)

## Decision Tree 결과
| 아이디어 | 사용자 가치 | 기술 실현성 | 시장 차별성 | 비즈니스 모델 | 판정 |
|----------|------------|------------|------------|-------------|------|
| ... | ✅ `<근거 1줄>` / ❌ `<이유 1줄>` / ⚠️ 검증 필요 | (동일) | (동일) | (동일) | 승인/조건부/기각 |

> ✅/❌만 적지 마라. 근거 한 줄 또는 "⚠️ 검증 필요"를 함께 적는다 (할루시네이션 가드 §1·§4 강제).

## 착수 조건 (해당 시)
| 아이디어 | 선행 조건 | 착수 시점 |
|---------|---------|---------|
| ... | ... | ... |

## 실행 계획 (해당 시)
(단계별 실행 항목, 우선순위, 예상 일정)

## 인터뷰 항목 (해당 시)
(검증을 위한 인터뷰 질문, 대상, 확인 사항)

## 누락된 관점 (해당 시)
(토론에서 다루지 못한 관점, 후속 브레인스토밍 주제)

## 사용자 결정 대기
(사용자에게 최종 판단을 요청하는 질문)
```
