# AI 기초 사용 가이드

`ai-setup-guide.md` 설치를 마친 뒤 읽는다. 기본 사용을 익힌 다음 `ai-usage-guide.md`에서 저장소의 스킬, 규칙, 검증 흐름을 학습한다.

## 기본 인터페이스

Claude Code 터미널 또는 Conductor 워크스페이스에서 자연어 요청과 저장소 스킬을 사용할 수 있다.

### 자연어 요청

```text
이 프로젝트의 목적을 README.md 근거로 설명해줘.
Student 도메인을 스키마부터 화면까지 추적해줘.
최근 커밋 10개의 제목만 보여줘.
```

정확도가 필요한 요청에는 범위, 기준 파일, 원하는 결과를 함께 적는다.

### 슬래시 스킬

```text
/biz status       사업 현황과 우선 조치 확인
/test             전체 테스트 실행과 결과 요약
/pre-pr           변경 영역별 reviewer 실행
/commit           diff 분석 후 확인을 거쳐 커밋
/sdd 0            승인 가능한 신규 작업 선택
```

`/` 입력 시 현재 설치된 스킬을 확인한다. 문서에 없는 이름을 추측해 실행하지 않는다.

## 도구 승인 이해하기

AI가 파일을 수정하거나 명령을 실행할 때 승인 요청이 나타날 수 있다. 버튼 이름은 도구 버전에 따라 달라질 수 있지만 판단 기준은 같다.

| 작업 | 확인할 내용 | 기본 판단 |
| --- | --- | --- |
| 소스와 문서 읽기 | 경로에 비밀값이나 개인정보가 있는가 | 일반 파일만 허용 |
| 파일 편집 | 대상 파일, diff, 다른 사람 변경과 충돌 여부 | 범위가 맞을 때 허용 |
| 명령 실행 | 쓰기, 삭제, 네트워크, 운영 영향 | 명령을 이해한 뒤 허용 |
| 반복 자동 허용 | 허용 범위와 명령 접두사가 충분히 좁은가 | 익숙해진 뒤 제한적으로 사용 |

잘 모르겠으면 거부하고 명령의 목적과 영향을 설명해 달라고 요청한다. 거부해도 파일이나 Git 상태가 자동으로 손상되지 않는다.

다음 작업은 별도 확인 없이 허용하지 않는다.

- `rm`, `git reset --hard`, 대량 파일 덮어쓰기
- 운영 DB 변경, 프로덕션 배포, 결제나 외부 메시지 발송
- `.env`, 토큰, SSH 키, 사용자 개인정보 읽기 또는 출력
- 다른 사람이 만든 변경을 되돌리는 명령

## 첫 대화 연습

처음에는 읽기 전용 요청으로 시작한다.

### 1. 프로젝트 파악

```text
README.md와 .claude/CLAUDE.md를 읽고 이 저장소의 목적, 앱, 패키지를 표로 설명해줘.
```

설명에 실제 파일 경로가 포함됐는지 확인한다.

### 2. 현재 사업 상태 확인

```text
/biz status
```

현재 수치는 `docs/business/STATUS.md`, 정의는 `docs/business/5_metrics/metrics.md`를 기준으로 봐야 한다. WVO와 MAO는 단체 단위, MAU는 사용자 단위다. WVO가 미측정이거나 MAU 최신값이 없으면 임의로 계산하지 않는다.

### 3. 파일 탐색

```text
docs/business/ 폴더 구조와 각 문서의 단일 출처 역할을 설명해줘.
```

### 4. 스킬 확인

```text
/help
```

설치된 스킬과 기본 Claude Code 명령을 구분해 확인한다.

## 좋은 요청 만들기

| 모호한 요청 | 검증 가능한 요청 |
| --- | --- |
| 코드 고쳐줘 | StudentListPage 검색 실패를 재현하고 원인만 분석해줘 |
| 콘텐츠 만들어 | 현재 STATUS와 콘텐츠 운영 게이트를 먼저 확인해줘 |
| 다 해줘 | API 변경안을 먼저 제시하고 승인 전에는 편집하지 마 |
| 지표 알려줘 | STATUS 기준일과 metrics 정의를 함께 인용해줘 |

핵심 원칙:

1. 대상 파일과 문제를 구체적으로 지정한다.
2. 분석, 수정, 검증 중 원하는 단계와 권한을 밝힌다.
3. 현재 상태는 SSoT 파일을 지정한다.
4. AI가 모르는 외부 상태는 `미확인`으로 남기게 한다.
5. 중요한 판단은 근거 파일, 명령 결과, diff를 직접 확인한다.

## 역할별 연습

### 프로덕트 엔지니어

```text
Student 도메인을 schema.prisma, shared schema, API application/presentation,
Web feature/page 순서로 추적하고 실제 경로를 제시해줘.
```

```text
/test
```

테스트 실패 시 바로 수정시키기 전에 실패 명령, 첫 원인, 재현 범위를 요약하게 한다.

### 콘텐츠 마케터

```text
docs/business/STATUS.md와 docs/business/3_gtm/instagram.md를 읽고
현재 허용된 인스타 운영 범위만 설명해줘.
```

현재 사업 문서상 Instagram 콘텐츠 생산은 별도 재개 승인 전까지 동결되어 있다. 2026년 9월 90일 사업 검증 모드 채택은 재개 승인이 아니다. 기존 콘텐츠 열람과 수신 DM 대응은 가능하지만, 게시물 초안 생성이나 발행 재개는 게이트 재평가 없이 진행하지 않는다.

```text
docs/content/README.md와 docs/content/instagram/의 기존 산출물 구조만 정리해줘.
```

## 자주 하는 실수

### 답이 근거 없이 단정적임

```text
방금 답변의 각 주장에 실제 파일 경로와 줄 근거를 붙여서 다시 검증해줘.
확인할 수 없는 외부 상태는 미확인으로 표시해줘.
```

### AI가 범위를 넓힘

```text
지금은 원인 분석만 해줘. 파일 수정, 커밋, 외부 작업은 하지 마.
```

### 파일을 잘못 수정함

먼저 `git diff -- <파일>`로 변경을 확인한다. 사용자 변경과 AI 변경이 섞였으면 파일 전체를 되돌리지 말고 잘못된 줄만 복구한다.

### 한 번에 너무 많은 작업을 함

```text
1단계 결과와 검증만 보여줘. 내가 확인한 뒤 다음 단계로 가자.
```

### 이전 대화를 기억한다고 가정함

세션이 바뀌면 필요한 근거를 다시 제시한다. 장기 작업은 `/save-session`, `/resume-session`을 사용하고 현재 Git 상태를 다시 확인한다.
