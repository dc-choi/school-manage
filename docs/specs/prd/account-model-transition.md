# PRD: 계정 모델 전환 (공유 → 개인)

> SDD 작성자가 작성하는 제품 요구사항 문서입니다.
> 사업 에이전트의 `docs/business/` 문서를 기반으로 작성합니다.

## 배경/문제 요약

- 참고: `docs/brainstorm/2026-02-26.md` (GTM 전략), `docs/brainstorm/2026-02-21.md` (수익화 모델), `docs/brainstorm/2026-02-23.md` (행사 메모 카드)
- 문제:
  - **공유 계정 구조가 구조적 이탈을 유발한다.** 현재 하나의 계정(예: "중고등부")을 여러 교리교사가 공유 사용. 신당동/금호동에서 실증: 2명 이상 교리교사가 개별 가입 시 "누가 어떤 계정을 쓰는지" 혼란 발생 → 사용 중단 (A유형 이탈)
  - **개인 식별 불가.** 교리교사 개인별 활동 추적, 권한 분리, 보고 기능 구현이 불가능
  - **IDOR 취약점의 근본 원인.** 계정-리소스 소유권 관계가 1:N 직결 구조라 수평 권한 상승 위험
  - **교구/본당/조직 계층이 없다.** 현실의 가톨릭 구조(교구 → 본당 → 초등부/중고등부 → 학년반)를 반영할 엔티티가 없음
  - **2단계 기능의 공통 선행 조건.** 행사 메모 카드, 사제 보고용 공유 링크, 구독 모델 모두 Parish/Church/Organization 엔티티를 전제
- 현재 상태:
  - Account(공유) → Group(1:N) → Student(1:N) → Attendance(1:N) 구조
  - 38개 가입 계정 (03-12 기준), WAU 8곳, 26개+ 본당 추적 중
  - 금호동: 2개 계정 생성 후 혼란, 신당동: 구조 불일치로 미사용 지속
- 목표 상태:
  - Parish(교구) → Church(본당) → Organization(조직: 초등부/중고등부) → Group(학년/부서) ↔ Student(N:M) → Attendance
  - Account(개인)는 Organization 소속 (조직별 1개 계정 발급, 멤버십 테이블 불필요)
  - Student-Group N:M: 학생이 여러 Group에 동시 소속 (예: "중1반" + "복사단")
  - 교리교사별 개인 계정 → 본당(Church)/조직 합류 → 관리자 승인 → 조직 내 Group/Student/Attendance 접근
  - Parish(교구) 데이터: 마이그레이션 시 한국 가톨릭 교구 전체 사전 적재
  - 개인 사용자(조직 미소속): 마이그레이션에서 제외, 재로그인 시 교구→본당→조직 선택 필요
  - 기존 데이터 무결성 100% 유지 마이그레이션

## 목표/성공 기준

- **목표**: 공유 계정을 개인 계정 + Parish/Church/Organization 계층 구조로 전환하여 구조적 이탈 원인 제거 및 2단계 기능의 기반 확보
- **성공 지표**:
  - WAU 12명 이상 달성 (현재 7명 → 70% 이상 증가)
  - A유형 본당(신당동/금호동) 재접촉 후 활성화 전환 1곳 이상
  - 신규 가입자 "본당(Church) 합류" 플로우 완료율 70% 이상
  - 마이그레이션 후 데이터 손실 0건
  - IDOR 취약점 구조적 해소 (Account.organizationId 기반 소유권 체인)
- **측정 기간**: 배포 후 4주 (5월 중)

## 사용자/대상

- **주요 사용자**: 주일학교 교리교사 (개인)
- **사용 맥락**:
  - **신규 가입 (새 조직 생성)**: 개인 계정 생성 → 교구 선택 → 본당(Church) 검색/생성 → 조직(Organization) 생성 → 관리자로 즉시 진입
  - **신규 가입 (기존 조직 합류)**: 개인 계정 생성 → 본당/조직 검색 → 합류 요청 → 관리자 승인 대기 → 승인 후 접근
  - **기존 사용자 (조직 계정)**: 기존 ID/PW로 로그인 → 마이그레이션 후 개인 계정 유지 + 기존 데이터가 Parish → Church → Organization 하위로 자동 전환 + 관리자 역할 부여
  - **기존 사용자 (개인 계정)**: 기존 ID/PW로 로그인 → 교구→본당→조직 선택 화면으로 유도
  - **다수 교리교사 본당**: 첫 번째 교리교사 = 관리자, 이후 교리교사 = 합류 요청 → 관리자 승인 후 접근

## 범위

### 포함

- Account 모델 변경: 공유 계정 → 개인 계정 (로그인 단위 = 개인)
- Parish(교구) 엔티티 신설: 최상위 교구 단위 (예: 서울대교구)
- Church(본당) 엔티티 신설: Parish 하위 본당 (예: 장위동 성당)
- Organization(조직) 엔티티 신설: Church 하위 단체 (초등부, 중고등부 등)
- Account-Organization 직접 관계: 조직별 1개 계정 발급 (Account.organizationId FK), 역할(관리자/교사) 구분
- 조직 합류 승인제: 기존 조직에 합류 시 관리자 승인 필요 (JoinRequest 엔티티)
- Group 소유권 이전: Account 직속 → Organization 하위 (organizationId)
- Student-Group N:M 관계: 학생이 여러 Group에 동시 소속 가능 (학년/부서 두 축 그룹핑)
- 마이그레이션: Parish(교구) 16개 사전 적재 + 조직 계정(데이터 있음)은 본당별 교구 매핑 후 Church/Organization 자동 생성 + 관리자 역할. 미사용 계정(데이터 없음)은 조직 미소속 유지 → 재로그인 시 /join 화면에서 합류 또는 생성
- 인증 플로우 변경: 로그인 후 본당(Church) 선택/합류 → 조직 선택 → (새 조직: 즉시 진입 / 기존 조직: 승인 대기)
- IDOR 구조적 해소: Account.organizationId 기반 소유권 검증
- 마이그레이션 공지: 기존 사용자에게 사전 안내

### 제외

- Refresh Token 구현 (별도 SDD)
- Parish/Church/Organization 간 데이터 이동/공유
- 결제/구독 기능 (3단계)
- 행사 메모 카드 (별도 SDD, 수요 검증 후)
- 사제 보고용 공유 링크 (별도 SDD)
- 교리교사 역할/권한 세분화 (3단계)
- Church 내 다중 Organization 관리 고도화 (3단계)

## 사용자 시나리오

1. **신규 교리교사 (새 조직)**: 회원가입 → 개인정보 동의 → 교구 선택 → 본당(Church) 검색/생성 → 조직(Organization) 생성(예: 중고등부) → 관리자(admin)로 즉시 대시보드 진입
2. **신규 교리교사 (기존 조직 합류)**: 회원가입 → 개인정보 동의 → 본당/조직 검색 → 합류 요청 → "승인 대기" 화면 → 관리자 승인 후 대시보드 진입
3. **기존 사용자 (조직 계정) 마이그레이션**: 기존 ID/PW로 로그인 → 기존 데이터가 Parish + Church + Organization 하위로 자동 전환 + 본인은 관리자 → 기존 데이터 그대로 사용
4. **기존 사용자 (개인 계정)**: 기존 ID/PW로 로그인 → organizationId 없음 → /join 화면에서 교구→본당→조직 선택
5. **같은 본당 두 번째 교리교사**: 개인 계정 생성 → 본당/조직 검색 → 합류 요청 → 기존 관리자가 승인 → 동일 Group/Student/Attendance 데이터 접근
6. **관리자 승인 처리**: 관리자 로그인 → 대시보드에서 합류 요청 알림 → 승인/거절 → 승인 시 요청자 Account.organizationId 설정
7. **여러 조직 교리교사**: 조직별 별도 계정 생성 → 각 계정으로 로그인하여 해당 조직 데이터 접근

## 요구사항

### 필수 (Must)

- [ ] Account는 조직별 1개 발급 (같은 사람이 2개 조직이면 2개 계정)
- [ ] Parish(교구) 엔티티 신설 + 한국 가톨릭 교구 데이터 사전 적재
- [ ] Church(본당) 엔티티 신설 (이름, 생성일, 삭제일, parishId) + 사용자 생성 가능
- [ ] Organization(조직) 엔티티 신설 (이름, 생성일, 삭제일, churchId) + 사용자 생성 가능
- [ ] Account.organizationId FK + role (관리자/교사) 추가
- [ ] JoinRequest(합류 요청) 엔티티 신설 (기존 조직 합류 시 관리자 승인 필요)
- [ ] AccountSnapshot 엔티티 신설 (계정 히스토리 추적)
- [ ] Group은 Organization 하위로 이전 (accountId → organizationId)
- [ ] Student-Group N:M 관계 (기존 1:N → 중간 테이블로 전환, 학년/부서 두 축 그룹핑)
- [ ] 로그인 후 교구→본당→조직 선택/합류 플로우
- [ ] 조직 생성 시 관리자(admin) 자동 부여, 기존 조직 합류 시 관리자 승인 후 교사(teacher) 부여
- [ ] 기존 데이터 마이그레이션 (데이터 손실 0건)
- [ ] 조직 계정(데이터 있음) → 본당별 교구 매핑 후 Church + Organization 자동 생성 + 관리자 역할
- [ ] 미사용 계정(데이터 없음) → 조직 미소속 유지, 재로그인 시 /join에서 기존 조직 합류 또는 신규 생성
- [ ] 멤버십 기반 소유권 검증 (IDOR 구조적 해소)
- [ ] 마이그레이션 사전 공지 (기존 사용자 대상)

### 선택 (Should)

- [ ] 본당(Church) 검색 (이름 기반)
- [ ] 조직 멤버 목록 조회
- [ ] 합류 요청 알림 (관리자 대시보드)

### 제외 (Out)

- Parish/Church/Organization 삭제 (소프트 삭제만, 관리자만)
- 멤버 강제 퇴장
- Parish/Church/Organization 간 데이터 병합
- 멤버별 Group 접근 권한 세분화 (조직 내 전체 접근)

## 제약/가정/리스크

- **제약**:
  - 1인 개발, 주 25시간 (Track A 15시간)
  - MySQL + Prisma ORM (기존 스택 유지)
  - 기존 19개 본당 데이터 무결성 보장 필수
- **가정**:
  - 교리교사는 개인 계정에 거부감이 없다 (현재도 개인 ID/PW로 로그인)
  - Organization 합류 플로우가 추가되어도 온보딩 이탈이 크지 않다 (기존 그룹 만들기와 난이도 유사)
  - 관리자 승인이 지연되어도 사용자 이탈이 크지 않다 (교리교사 간 오프라인 소통 가능)
  - A유형(신당동/금호동) 이탈은 계정 모델이 근본 원인이며, 전환 후 재접촉 시 활성화된다
- **리스크**:
  - 마이그레이션 실패 시 기존 사용자 데이터 손실 → 완화: 사전 백업 + 장위동 우선 실행 + 수동 검증
  - 기존 사용자 혼란 (로그인 방식 변경) → 완화: 마이그레이션 공지 3항목 + 직접 연락처 제공
  - SDD + 구현 일정 도전적 (3월 SDD, 4월 구현) → 완화: MVP 범위 최소화

## 롤아웃/검증

- **출시 단계**:
  1. 장위동 마이그레이션 (4년치 데이터, 스크립트 + 수동 검증, 사전 통화 필수)
  2. 정릉4동 마이그레이션 (스크립트 자동화)
  3. 사용 중 본당(가재울/창동/길음동/흑석동/창5동) — 데이터 적음, 재입력 권고
  4. 도입 상태 본당 — 재입력 권고
  5. 신규 가입은 전환 후 즉시 개인 계정 + Organization 합류 플로우
- **이벤트/로그**:
  - `church_created`: 본당 생성
  - `organization_created`: 조직 생성 (관리자로 즉시 진입)
  - `join_requested`: 조직 합류 요청
  - `join_approved`: 합류 요청 승인
  - `join_rejected`: 합류 요청 거절
  - `migration_completed`: 기존 계정 마이그레이션 완료
- **검증 방법**:
  - 마이그레이션 전후 데이터 정합성 비교 (Group/Student/Attendance 건수 일치)
  - 신규 가입 → Church/Organization 합류 플로우 완료율 측정
  - A유형 본당(신당동/금호동) 재접촉 후 활성화 여부 확인
  - WAU 12명 달성 추적 (5월 목표)
