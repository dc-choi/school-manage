# 역할별 인재상과 선발 기준

향후 채용이나 외부 협업 기회가 생겼을 때 빠르게 판단하고 온보딩하기 위한 사전 설계 문서다.
현재 채용 공고, 충원 확정, 계약 조건을 뜻하지 않는다. 역할별 4주 실행 계획은
`product-engineer.md`, `content-marketer.md`를 따른다.
아래 미들급 연차 환산과 4주 기대 수준은 미래 역할 설계 기준이며, 현재 모집, 면접, 오퍼 또는 예정 인원을 뜻하지 않는다.

## 사용 원칙

- 인재상은 완성된 정답이 아니라 현재 사업 단계에 맞춘 초기 가설이다.
- 학력, 나이, 성별, 종교 같은 직무 외 특성을 역량의 대리 지표로 사용하지 않는다.
- 가톨릭 신앙 여부가 아니라 도메인 용어와 사용자 맥락을 존중하고 학습하는 능력을 본다.
- 필수 기준과 선호 기준을 구분하고, 모든 판단에는 관찰한 행동이나 산출물 근거를 남긴다.
- 후보 평가에서 확인한 강점과 공백을 Week 1 학습 순서와 첫 과제에 연결한다.

## 기준 레벨

| 역할 | 기준 레벨 | 연차 환산 | 4주 후 기대 수준 |
| --- | --- | ---: | --- |
| 프로덕트 엔지니어 | 실무형 미들 | 3~5년차급 | 승인된 변경을 설계, 구현, 검증하고 감독하에 배포와 복구 수행 |
| 콘텐츠 마케터 | 실무형 미들 | 2~4년차급 | 운영 게이트 안에서 콘텐츠, 수신 DM, 지표 보고를 독립 수행 |

- 연차는 기대 수준을 설명하는 기준이지 자동 합격이나 탈락 조건이 아니다.
- 연차 범위 밖이어도 포트폴리오와 선발 과제로 동등한 수행 능력을 증명하면 같은 기준으로 판단한다.
- 이 문서의 4주 계획은 미들급을 전제로 한다. 주니어 또는 시니어로 목표를 바꾸면 기간, 과제,
  대표 리뷰 범위와 역할 책임을 함께 다시 설계한다.

## 공통 인재상

| 기준 | 기대 행동 | 확인 근거 |
| --- | --- | --- |
| 사용자 중심 | 기능이나 콘텐츠보다 교리교사의 실제 주일 업무를 먼저 이해 | 기존 제품 또는 콘텐츠를 사용자 언어로 설명 |
| 근거 중심 | 수치, 코드, 외부 상태를 추측하지 않고 출처와 한계를 표시 | 모르는 내용을 미확인으로 남긴 사례 |
| 주도성과 경계 | 맡은 결과는 끝까지 추적하되 권한 밖 결정은 즉시 확인 | 독립 수행과 에스컬레이션 경험 |
| 작은 단위 실행 | 큰 계획을 검증 가능한 첫 산출물로 줄여 완료 | 짧은 주기의 배포, 실험, 개선 사례 |
| 문서 협업 | 비동기 환경에서 결정, 진행, 예외를 다른 사람이 이어받게 기록 | PR, 작업 문서, 회고 등 실제 산출물 |
| AI 활용 판단 | AI를 빠르게 사용하지만 출력, 권한, 개인정보를 직접 검증 | AI 오류를 발견하거나 범위를 제한한 사례 |
| 보안과 신뢰 | 최소 권한과 개인정보 비노출을 속도보다 우선 | 민감 작업의 승인 또는 회수 경험 |

선발 결과를 합격 여부로만 남기지 않는다. 합류 전 아래 내용을 한 장으로 기록한다.

| 항목 | 기록 내용 |
| --- | --- |
| 역할 결과 | 첫 4주에 독립 수행할 결과 한 가지 |
| 확인된 강점 | 바로 위임할 수 있는 역량과 근거 |
| 학습 공백 | Week 1에 보강할 기술, 도메인, 도구 |

문제가 아니라 인재상, 과제 난이도, 지원 설계 중 무엇이 잘못됐는지 구분해 이 문서를 갱신한다.

## 프로덕트 엔지니어

### 역할 미션

사용자 가치와 운영 안정성을 함께 이해하고, 승인된 범위의 제품 변경을 설계부터 검증,
배포, 복구까지 이어받는다. 많은 기능을 만드는 것보다 현재 우선순위를 정확히 고르는 것이 먼저다.

### 필수 기준

| 역량 | 기대 수준 | 확인 방법 |
| --- | --- | --- |
| 코드 추적 | TypeScript 풀스택 흐름을 모르는 코드에서도 끝까지 추적 | Student 흐름을 스키마부터 화면까지 설명 |
| 검증 습관 | 수정 전에 재현하고 테스트, diff, 로그로 결과 확인 | 작은 버그의 원인 분석과 검증 계획 |
| 운영 안전 | 배포, DB 변경, 롤백의 위험과 승인 경계 이해 | 장애 상황에서 중단, 보고, 복구 순서 설명 |
| 설계 판단 | 기존 패턴을 우선하고 불필요한 구조 확장을 피함 | 대안의 비용과 선택 이유 설명 |
| 사업 연결 | STATUS와 TARGET을 읽고 보류 항목을 임의 착수하지 않음 | 기술 과제의 사용자 가치와 진입 조건 설명 |
| 인수 가능성 | 코드뿐 아니라 반복 절차와 예외를 문서화 | 다른 사람이 이어받을 수 있는 PR 또는 런북 |

### 선호 기준

- React, Express, tRPC, Prisma, MySQL 또는 인접 스택 운영 경험
- 작은 제품이나 제한된 인력 환경에서 우선순위를 조정한 경험
- 개인정보를 다루는 서비스의 권한, 로그, 마이그레이션 경험
- AI 코딩 도구를 검증 가능한 개발 흐름에 연결한 경험

### 주의 신호

- 재현과 테스트 없이 바로 코드를 고치거나 AI 출력을 그대로 반영한다.
- 현재 사업 근거보다 새 기술이나 대규모 재설계를 우선한다.
- 초기부터 운영 DB, secrets, 프로덕션 전체 권한을 요구한다.
- 실패, 불확실성, 롤백 계획을 기록하지 않는다.

### 선발 확인 과제

1. 읽기 전용으로 Student 도메인 흐름을 추적하고 불확실한 부분을 표시한다.
2. 작은 버그 사례에서 재현, 원인, 최소 수정, 테스트 계획을 제시한다.
3. API 배포 실패 상황에서 영향 확인, 보고, 롤백 순서를 설명한다.

실제 서비스에 반영할 산출물을 요구한다면 범위와 보상을 별도로 합의한다. 후보에게 운영 자격
증명이나 실제 개인정보를 제공하지 않는다.

## 콘텐츠 마케터

### 역할 미션

제품의 실제 동작과 사용자 목소리를 외부 메시지로 정확히 번역하고, 승인된 운영 상태 안에서
콘텐츠, 수신 DM, 지표 기록을 일관되게 운영한다. 발행량보다 신뢰와 학습 가능성이 먼저다.

### 필수 기준

| 역량 | 기대 수준 | 확인 방법 |
| --- | --- | --- |
| 사용자 언어 | 기능명을 현장 결과와 교리교사 언어로 전환 | 기존 기능을 한 문장과 카드 구성으로 설명 |
| 사실 검증 | 제품 동작, 수치, 시장 주장을 출처와 함께 확인 | 기존 콘텐츠의 과장이나 미확인 주장 교정 |
| 운영 규율 | 동결, 승인, 공개 범위를 콘텐츠 욕심보다 우선 | 게시할 수 없는 상황의 대체 과제 제시 |
| 지표 해석 | 기간, 분모, 사용자와 단체 단위를 구분 | 도달, 클릭, 가입 데이터를 과장 없이 해석 |
| 채널 대응 | DM을 지원, 영업, 법률, 개인정보 사안으로 구분 | 예시 DM의 답변과 이관 기준 작성 |
| 개인정보 보호 | 스크린샷과 AI 입력에서 사용자 정보를 제거 | 안전한 에셋 검수 절차 설명 |

### 선호 기준

- SaaS, 커뮤니티, 비영리 또는 현장 운영자를 대상으로 한 콘텐츠 경험
- 인터뷰 원문을 왜곡하지 않고 카드뉴스나 사례로 편집한 경험
- Instagram Insights, UTM, 퍼널 데이터를 같은 기간으로 비교한 경험
- 가톨릭 용어와 교리교사 환경을 존중하며 빠르게 학습할 수 있는 역량

### 주의 신호

- 검증되지 않은 시장 1위, 유일, 전환 성과 같은 표현을 쉽게 사용한다.
- 운영 게이트와 제품 준비도보다 게시 빈도와 팔로워 수를 우선한다.
- 공용 계정 비밀번호나 개인정보를 일반 문서와 AI 도구에 입력한다.
- 클릭, 가입, MAO처럼 단위가 다른 지표를 하나의 전환율로 계산한다.

### 선발 확인 과제

1. 기존 콘텐츠 한 건의 제품 사실, 톤, 개인정보, CTA를 검수한다.
2. 수신 DM 사례를 직접 답변, 대표 이관, 엔지니어 이관으로 구분한다.
3. 서로 다른 기간의 Insights 표를 보고 확정할 수 있는 것과 없는 것을 나눈다.

## 평가 척도

| 점수 | 의미 |
| ---: | --- |
| 1 | 근거가 없거나 현재 역할 경계와 충돌 |
| 2 | 구체적 안내와 잦은 리뷰 아래 수행 가능 |
| 3 | 합의된 범위에서 독립 수행 가능 |
| 4 | 기준을 개선하고 다른 사람에게 전수 가능 |

- 필수 기준은 평균 점수로 상쇄하지 않는다. 보안, 사실 검증, 권한 경계의 중대한 주의 신호가
  있으면 다른 강점과 별도로 확인한다.
- 선호 기준 부족은 자동 탈락 사유가 아니다. 4주 안에 보완 가능한지 온보딩 비용으로 판단한다.
- 최종 판단에는 점수와 함께 근거 산출물, 확인하지 못한 항목, 온보딩 보완 계획을 기록한다.
