# PRD: 학생 등록 중복 확인

> 상태: Draft | 작성일: 2026-04-30

## 배경/문제 요약

- 참고: `docs/specs/README.md` TARGET FUNCTIONAL P2 — "학생 등록 중복 확인", 피드백 `docs/business/0_feedback/entries/2026-04-01-jangwidong.md`, 카테고리 `docs/business/0_feedback/feedback-categories.md` "데이터 관리"
- **문제**:
    - 팀 사용(복수 교사) 환경에서 **다른 교사가 이미 등록한 학생을 모르고 다시 등록하는 사례**가 발생한다. 장위동초(04-01) 학생 25→51명(+26) 대량 등록 중 직접 보고됨.
    - 단건 등록(`student.create`)·일괄 등록(`student.bulkCreate`) 모두 **중복 검출 로직이 없다**. 동일 조직 내 같은 이름·세례명 학생이 그대로 추가된다.
    - 잠재 영향 범위: 우면동(13계정)·사창동초(11계정)·후곡(12계정) 등 **다중 교사 운영 본당** 12곳 이상 (STATUS.md 04-27 기준 계정 5+ 본당).
- **현재 상태**:
    - `CreateStudentUseCase.execute` (`apps/api/src/domains/student/application/create-student.usecase.ts`): 권한 검증 후 곧바로 `student.create`. 동일 조직 내 동명/동세례명 검사 없음.
    - `BulkCreateStudentsUseCase.execute` (`apps/api/src/domains/student/application/bulk-create-students.usecase.ts`): 트랜잭션 내에서 입력 배열을 순회하며 `student.create` 호출. 입력 배열 내부 중복도 검증하지 않음.
    - DB 스키마(`apps/api/prisma/schema.prisma` Student 모델): `society_name varchar(50)`, `catholic_name varchar(50)` — UNIQUE 제약 없음, 동명이인 허용 설계.
    - 단건 등록 폼(`StudentCreateForm`)·엑셀 Import 모달(`StudentImportModal`) 모두 클라이언트 사전 검증에서 중복을 체크하지 않음.
- **목표 상태**:
    - 단건/일괄 등록 시 **이름+세례명 정규화 동일** 학생이 같은 조직에 이미 존재하면 사용자에게 경고 + 확인 요청. 강제 등록 의사를 받은 경우에만 진행.
    - 일괄 등록은 **입력 배열 내부 중복** + **DB 기존 학생과의 중복** 양쪽 모두 검출.
    - DB UNIQUE 제약은 도입하지 않음(동명이인 운영 사례 존재 — 본당당 같은 이름·세례명 학생 가능). 애플리케이션 레벨 경고가 적합.

## 목표/성공 기준

- **목표**: 팀 사용 환경에서 **다른 교사가 이미 등록한 학생의 중복 등록**을 감지하고, 사용자가 인지한 상태에서만 등록되도록 한다.
- **성공 지표**:
    1. 동일 조직에 `societyName + catholicName` 정규화 동일 학생이 있을 때, 추가 등록 시 경고 응답(또는 확인 후 등록) 통합 테스트 통과 (단건 ≥ 2건, 일괄 ≥ 3건)
    2. 일괄 등록 시 입력 배열 내부 중복(이름+세례명 동일 2행 이상)을 사전에 감지 (≥ 2건)
    3. 정상 등록 회귀 테스트 0건 실패 (기존 사용자 시나리오 보호)
    4. 단발 측정: 머지 후 4주간 "중복 경고 후 강제 등록" 발생 비율 측정 — 기준치 없음(관찰 지표). GA4 이벤트로 수집.
- **측정 기간**: 머지 후 즉시 통합 테스트 + 4주 후 GA4 이벤트 회고

## 사용자/대상

- **주요 사용자**: 팀 사용 본당의 교리교사 (조직 관리자/교사 모두). 한 조직에 2계정 이상 운영하는 본당이 핵심.
- **사용 맥락**:
    - 학기 초/대량 등록 시기에 다수 교사가 분담 입력
    - 엑셀 Import에 누락된 학생을 단건 폼으로 보강
    - 다른 교사의 등록 결과를 모르는 상태에서 본인이 추가
- **부차 사용자**: 단일 운영자 본당. 본인이 이미 등록한 학생을 잊고 다시 추가하는 경우에도 동일 가치 제공.

## 범위

### 포함

- **단건 등록 경로** (`student.create`): 동일 조직 내 이름+세례명 정규화 동일 재학생 존재 검증.
- **일괄 등록 경로** (`student.bulkCreate`):
    - 입력 배열 내부 중복(같은 페이로드에서 이름+세례명 동일 2행 이상)
    - 입력 배열 vs DB 기존 학생 중복
- **응답 형식**: 중복 발견 시 `CONFLICT`가 아닌 **사용자가 강제 진행 가능한 형태**.
    - 기본안 (FD에서 확정): mutation에 `force` 플래그 추가. `force: false`(기본) + 중복 존재 → `CONFLICT` + 중복 학생 메타(이름·세례명·그룹·등록일) 반환. `force: true` → 검증 생략하고 등록.
- **클라이언트 UX**:
    - 단건 폼: 등록 버튼 클릭 → 서버 응답이 CONFLICT면 다이얼로그(이미 등록된 학생 정보 표시 + "그래도 등록" / "취소") → 강제 등록 시 `force: true` 재시도.
    - 엑셀 Import 모달: 미리보기 단계에서 사전 검증 호출(또는 등록 시점 검증) → 중복 학생을 행 단위로 마킹 + 일괄 강제/일괄 취소/개별 강제 선택.
- **비교 정책 (정규화)**:
    - 이름·세례명 모두 `trim()` + 다중 공백 단일 공백 변환 후 비교.
    - 세례명 NULL 처리: 기본안은 **세례명이 둘 다 NULL일 때만 매칭** (한쪽이 NULL이면 다른 사람으로 간주) — FD에서 정책 확정.
- **삭제·졸업 학생 처리**:
    - 기본 비교 대상: `deletedAt IS NULL`인 재학생만 (졸업 학생은 `graduatedAt` 있어도 deletedAt 없으면 포함, 동명이인 신규 입학 인지 가능하도록).
    - 삭제 학생은 비교 제외 (재등록 의도 명확).
- **GA4 이벤트**: 중복 경고 노출/강제 등록/취소 이벤트 추가 (4주 회고용).
- **통합 테스트**: 단건/일괄 각 시나리오 커버 (≥ 8건).

### 제외

- **DB 레벨 UNIQUE 제약**: 동명이인 운영 사례를 차단하므로 도입 안 함. account-name-unique와 다른 정책.
- **출석 등 다른 도메인의 중복 검출**: 본 작업은 학생 등록 한정.
- **학생 수정(`student.update`) 시 중복 검증**: 본 PRD 범위 외 — 수정 경로의 동일 정책은 후속 과제로.
- **연락처(`contact`/`parentContact`)·생년월일 기반 매칭**: 데이터 누락이 잦아 신뢰도 낮음. 본 PRD에서는 이름+세례명만 사용.
- **조직 간 비교**: 본 PRD는 같은 `organizationId` 스코프만 검사. 다조직 비교는 의미 없음.

## 사용자 시나리오

1. **팀 환경 단건 중복**: 교사 A가 "박민수/베드로"를 등록 후, 교사 B가 동일 학생을 추가 등록 시도 → 경고 다이얼로그("이미 등록된 학생: 박민수(베드로) — 1학년, 04-15 등록"). B가 "취소" 선택 → 등록되지 않음.
2. **팀 환경 강제 등록(동명이인)**: 같은 본당에 "이서연/안나" 2명이 실재 → 경고 후 "그래도 등록" 선택 → `force: true`로 재요청 → 등록 성공.
3. **일괄 등록 입력 내부 중복**: 엑셀 50행 중 "김지훈/요셉"이 2행에 있음 → 미리보기 단계에서 2개 행 마킹 + "이 중 1건만 등록" / "양쪽 모두 등록" 선택.
4. **일괄 등록 DB 기존 학생 중복**: 엑셀 50행 중 5명이 이미 DB에 존재 → 미리보기에서 5행 마킹 + 일괄 처리 옵션.
5. **세례명 미입력 매칭(기본안)**: A "정수민/-" 존재. B "정수민/-" 입력 → 둘 다 세례명 NULL이므로 매칭 → 경고. (FD에서 확정 정책 검토)
6. **세례명 다른 동명**: A "정수민/요한", B "정수민/마르코" → 세례명 다름 → 매칭 안 함 → 정상 등록.
7. **삭제된 학생 재등록**: 삭제(`deletedAt`)된 "박민수/베드로" 존재 → 신규 등록 시 매칭 제외 → 정상 등록 (재등록 의도).
8. **단일 운영자 부주의**: 1계정 본당에서도 본인이 잊은 등록 발생 → 경고 → 인지 후 취소.

## 제약/가정/리스크/의존성

- **제약**:
    - 기존 정상 등록 사용자 흐름 변경 금지 (중복 없을 때 기존과 동일하게 1회 호출로 완료).
    - `student.bulkCreate`는 단일 트랜잭션 — 검증과 등록 사이 race 가능성 인식. 검증 후 등록 사이의 짧은 시간에 다른 교사가 동일 학생을 등록한 경우는 **수용 가능 손실**(다음 회 등록 시 다시 경고됨, 데이터 정합성 영향 없음).
- **가정**:
    - 정규화는 `trim` + 공백 단일화로 충분(실제 데이터 관찰: `societyName`은 한글 짧은 이름이 다수 — 대소문자/유니코드 정규화 필요성 낮음).
    - 본당당 학생 수 ≤ 500명 (현 최대 위례성모 220 수준). 일괄 등록 시 DB 조회 비용 미미.
    - GA4 이벤트는 기존 측정 인프라 재사용.
- **리스크**:
    - **R1 (중)**: 정규화·세례명 NULL 정책이 사용자 기대와 다를 경우 오감지/누락. → FD에서 정책 확정 + 4주 GA4 회고에서 강제율로 검증.
    - **R2 (저)**: `force: true` 우회 의도가 단순 "닫기→재시도"로 학습되어 실질 차단 효과 미흡. → UX에서 충돌 학생 정보를 충분히 노출.
    - **R3 (저)**: 미리보기 단계 사전 검증 호출이 일괄 트랜잭션과 분리되어 race 가능 → 위 "수용 가능 손실"로 처리.
    - **R4 (저)**: 엑셀 Import 모달의 중복 행 마킹 UI 복잡도 증가 → FD에서 단순한 "행 색상 + 체크박스 토글" 수준으로 제한.
- **내부 의존성**: `apps/api/src/domains/student/application/{create,bulk-create}-students.usecase.ts`, `packages/shared/src/schemas/student.ts` (Zod 스키마에 `force` 추가), `apps/web` 학생 폼·Import 모달.
- **외부 의존성**: 없음.

## 오픈 이슈

- [ ] **세례명 NULL 비교 정책 확정** (FD): 둘 다 NULL일 때만 매칭 vs 한쪽 NULL이어도 societyName만으로 매칭. 기본안은 전자.
- [ ] **`force` 플래그 위치**: mutation 입력 최상위 vs 학생 항목별. 일괄에서 행 단위 강제 가능 여부와 직결. 기본안: 일괄은 **행 단위 force 플래그**(특정 행만 강제), 단건은 입력 최상위.

- [ ] **졸업 학생 매칭**: 졸업한 학생도 매칭 대상 vs 졸업 학생은 제외. 기본안: 졸업 학생 포함(같은 학생이 다시 입학한 경우는 동일 학생 식별이 의미 있음).
