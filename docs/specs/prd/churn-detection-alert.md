# PRD: 이탈 감지 자동 알림 (운영자용)

> SDD 작성자가 작성하는 제품 요구사항 문서입니다.
> 사업 에이전트의 `docs/business/` 문서를 기반으로 작성합니다.

## 배경/문제 요약

- 참고: `docs/brainstorm/2026-03-16/acquisition-retention.md` (아이디어 3)
- 참고: `docs/business/STATUS.md` (비활성/이탈 본당 목록)
- 문제: 2030세대 교리교사는 Silent Churn 특성이 강해 불만/이탈 시 피드백 없이 조용히 떠남. 흑석동(15일 미활동), 창동(23일), 월곡동(19일) 사례에서 운영자가 이탈을 인지한 시점에는 이미 복귀 가능성이 낮음
- 현재 상태: 운영자가 DB를 직접 조회해야 이탈을 파악 가능. 자동 감지/알림 수단 없음
- 목표 상태: 14일 미활동 단체를 자동 감지하여 운영자에게 이메일 알림 → 조기 개입으로 이탈 방지

## 목표/성공 기준

- **목표**: 이탈 위험 단체를 조기 감지하여 운영자가 선제 대응할 수 있도록 자동 알림 제공
- **성공 지표**: 알림 발송 후 7일 내 활동 재개율 (목표: 30%+)
- **측정 기간**: 출시 후 8주

## 사용자/대상

- **주요 사용자**: 서비스 운영자 (ADMIN_EMAIL 수신자)
- **사용 맥락**: 매일 아침 이메일 확인 시 이탈 위험 단체 파악 → 개별 연락/지원 결정

## 범위

### 포함

- 14일 미활동 단체 자동 감지 (출석 기록 기준)
- 전례력 예외 처리 (성주간/성삼일 등 미사 없는 기간 제외)
- 운영자 이메일 알림 (일간 요약)
- 알림 발송 이력 관리 (중복 방지)

### 제외

- 교리교사(ADMIN) 직접 알림 (브레인스토밍에서 인터뷰 선행 조건)
- 7일 미활동 경고 단계 (MVP 범위 축소, 14일 단일 기준)
- 방학 기간 자동 감지 (전례력 기반 예외만, 본당별 방학은 미지원)
- 알림 설정 UI (운영자 환경변수로 관리)

## 사용자 시나리오

1. **일간 알림 수신**: 운영자가 아침에 이메일 확인 → "이탈 위험 3곳: 흑석동(15일), A본당(14일), B본당(18일)" 확인 → 개별 연락 결정
2. **전례력 예외**: 성주간(월~토) 동안 미활동 단체 → 전례력 예외 기간이므로 알림 미발송
3. **활동 재개**: 알림 후 해당 단체가 출석 기록 → 다음 알림에서 자동 제외
4. **SMTP 미설정**: SMTP 환경변수 미설정 시 → 크론잡 실행되나 이메일 미발송 (기존 패턴 유지)

## 요구사항

### 필수 (Must)

- [ ] 매일 1회 크론잡으로 14일 이상 미활동 단체 감지
- [ ] 출석 기록 기준 마지막 활동일 산출 (출석이 한 번도 없는 단체 제외)
- [ ] 전례력 예외 기간(성주간/성삼일) 중에는 알림 미발송
- [ ] 운영자(ADMIN_EMAIL)에게 이탈 위험 단체 목록 이메일 발송
- [ ] 이메일에 단체명, 미활동 일수, 학생 수, 마지막 활동일 포함
- [ ] 동일 단체에 대해 7일 내 중복 알림 방지
- [ ] SMTP 미설정 시 graceful degradation (에러 없이 스킵)

### 선택 (Should)

- [ ] 알림 발송 로그 기록 (발송일, 대상 단체, 발송 결과)
- [ ] 알림 후 7일 내 활동 재개 여부 추적용 로그

### 제외 (Out)

- 교리교사 직접 알림/넛지
- 알림 임계값 설정 UI
- 방학 기간 본당별 커스텀 설정
- 푸시 알림/카카오톡 채널

## 제약/가정/리스크

- **제약**: 기존 SMTP 인프라(Nodemailer + Gmail) 재사용. 추가 외부 서비스 도입 없음
- **제약**: 기존 `node-schedule` 스케줄러 활용
- **가정**: 14일이 이탈 감지 적정 임계값 (흑석동 15일 사례 기반)
- **가정**: 출석 기록이 한 번도 없는 단체는 아직 온보딩 중이므로 이탈 대상 아님
- **리스크**: Gmail SMTP 일일 발송 한도 (500통/일) — 현 규모(66 모임)에서 문제없음
- **리스크**: 전례력 예외만으로 방학 기간 커버 불가 (여름/겨울 방학은 본당별 상이)

## 의존성

- **내부**: SMTP 인프라 (`apps/api/src/infrastructure/mail/`), 스케줄러 (`apps/api/src/infrastructure/scheduler/`), 전례력 계산 (`packages/utils/src/liturgical.ts`, `get-holydays.usecase.ts`)
- **외부**: Gmail SMTP (기존 설정)

## 롤아웃/검증

- **출시 단계**: 전체 배포 (운영자 1명 대상이므로 단계적 롤아웃 불필요)
- **이벤트/로그**: 크론잡 실행 로그, 이메일 발송 로그, 감지된 단체 목록 로그
- **검증 방법**: 알림 발송 후 7일 내 활동 재개율 추적 (DB 기준)

## 오픈 이슈

- [ ] 여름/겨울 방학 예외 처리 방안 (현재 전례력만 지원, 본당별 방학 미지원)
- [ ] 브레인스토밍 인터뷰 항목 미완료 (ADMIN 전원 vs 1명, 방학 시기, 이탈 원인)
