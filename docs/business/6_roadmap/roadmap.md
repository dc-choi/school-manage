# 로드맵

## 로드맵 원칙(가설/근거/검증)

### 가설

- 무료 핵심 완성 → 유저 확장 → 유료 멤버십 → 외부 연동 순서가 도입 장벽과 리스크를 최소화한다

### 근거

- 초기에는 무료 핵심으로 진입 장벽을 낮추고, 확산 후 유료 전환을 검증해야 한다는 합의
- 외부 연동(알림/결제/전자서명)은 비용/계약 리스크가 커서 후순위가 합리적이다

## 1단계(무료 핵심 완성) — 완료

### 가설

- 목표: 유저 피드백 기반으로 무료 핵심 기능 완성
- 범위
    > 완료. 세부 항목: `HISTORY.md` 참조

### 근거

- 반복 행정과 데이터 분산이 핵심 문제이며, 무료 핵심으로 주간 리듬을 먼저 안정화해야 도입 장벽이 낮다

## 3단계(베이직 플랜 — 알림/캘린더 시그니처) (2026-04-24 재편)

### 가설

- 목표: 베이직 플랜 완성. 수동 운영을 지원하는 유료 기능군으로 저가 구간 유료 전환 검증
- 범위
    - **알림/공지 발송 (시그니처)**: SMS/카톡 발송, 공지 리마인드. 사용량 제한 포함(예: 월 200건) + 초과분 애드온
    - **교사 공유 캘린더 (신규)**: 교리교사 일정 공유. 수동 이벤트 등록. 구글 캘린더 연동 검토 (OAuth)
    - 행사 운영: 일정/준비물/역할 배정, 참가자 명단/비상연락망 관리
    - 다음달 축일 관리 (이번달 축일은 2단계에서 완료)
    - **개발 인프라 고도화 (하네스 엔지니어링)** _(내부 인프라, 과금 대상 아님)_: 에이전트 출력 자동 리뷰, 에이전트 작업 Observability, 실패 시 자동 롤백+알림

### 근거

- 베이직 플랜은 "엑셀+카톡" 수동 운영을 대체하는 도구군 → 저가 구간 전환 장벽 최소화
- 알림은 외부 비용(SMS/카톡 건당)이 발생하지만 PMF 검증 시점에 유료 전환 트리거로 작동 가능
- 교사 공유 캘린더는 4단계 회의록 자동 연동의 전제 기능 (캘린더 먼저, 회의록 나중)

### 애드온 (플랜 독립)

- **NFC 출석 — 보류 (2026-04-24)**: 임팩트 불명확 + 하드웨어 검증 선행 필요. 기존 "2단계 파일럿 / 3단계 정식 출시" 항목은 **보류 상태로 전환**. PMF 검증 완료 후 재평가

## 4단계(프로 플랜 — 회의록 자동화 시그니처, 해자) (2026-04-24 재편)

### 가설

- 목표: 프로 플랜 완성. 회의록 자동화를 해자 기능으로 완성하여 프로 전환 및 락인
- 범위
    - **회의록 자동화 (시그니처, 해자)**: 단일 통합 시스템
        - 입력: 사용자 메모 직접 입력 OR 녹음 → Clova/Daglo 텍스트 추출
        - AI 분석: 미사/교리/공지/안건 자동 분류·정리 (Claude API 등)
        - PDF 생성 및 사용자 컨펌 (수정 가능, OK 시 다운로드)
        - 공지 섹션 날짜 파싱 → 3단계 교사 공유 캘린더에 자동 등록 (프로 전용 확장)
    - 등록 관리: 연/학기 등록제, 등록 여부/보상 기준 관리 (전자서명 연동)
    - 시상/보상 기준 자동 산출 (등록제 연계)
    - 개인정보 제공 동의서/참가 동의서 (**전자서명 연동** — 2단계 수동 체크에서 업그레이드)
    - 결제 모듈: 행사비/교재비

### 근거

- 회의록 자동화는 교사 주간 필수 업무를 대체 → **전환 비용(switching cost) 발생 → 구조적 해자** (상세: `2_bm/bm.md` 해자 가설)
- 기존 가치 1-5가 "편의성"에 머무르는 반면, 회의록 자동화는 **필수 업무 대체**로 이탈 비용 발생
- AI 비용(Clova/Daglo + Claude API)이 프로 가격(15,000원/월) 정당화 근거
- 외부 연동(전자서명/결제)은 비용/계약 리스크가 크며, 프로 전환 후 확장해야 한다

### 양업시스템 연동 사업 합의 체크리스트 (2026-05-11 신설, 단계 독립 의존성)

> 배경: 서울대교구 양업시스템 데이터 연동은 본당 도입 장벽 해소의 잠재 핵심 인프라. 2단계 범위에서 분리하여 단계 독립 의존성으로 격상. 사업 합의 → 사양 확정 → 구현의 체인 의존성이라 PG 체크리스트와 동일 패턴으로 관리. 합의 단계 진척에 따라 구현 단계(2-4단계 중 어디로 진입할지)는 추후 결정.

### 근거

- 인증/권한과 데이터 보존이 불안정하면 유료 기능 품질이 하락한다
- 외부 연동(SMS/카톡/전자서명/AI)은 계약/비용 확정 없이는 검증이 불가능하다
- 회의록 AI 파이프라인은 신규 의존성 — 사전 POC 없이 착수 시 비용/품질 리스크 과다
- PG·통신판매업은 체인 의존(단계 건너뛰기 불가)이므로 3단계 착수 최소 3-5주 전 시작 필요

## 확장 단계: 중개 플랫폼 전환 (미래)

### 가설

- 제품 성숙 후 "버스 대절·장소 연결" 중개 플랫폼으로 수익원 다변화 — 단, 법적·재무적 구조 재설계 필요
- 전환 타이밍: 3단계 베이직 PMF 확인 + 4단계 프로 해자 검증 완료 후 (현 시점 로드맵 외 장기 옵션)

### 필수 선행 조건

- **법적 검토**: 관광진흥법상 여행업 등록 요건 자문 (상세: `4_risk/risks.md` R22)
- **업종 지위 전환**: 525101(소매) → 525103(통신판매 중개업) 주업종 비중 이동
- **매출 인식 재설계**: GMV(총거래액) → 수수료 매출로 회계 전환, 세무사 감수
- **정산 시스템**: 공급자(버스/장소)에게 수수료 공제 후 정산하는 로직 + 세금계산서 역발행 대응
- **약관 개편**: "중개자 책임 한계" 조항 명시로 연대 책임 리스크 감쇄

### 근거

- 단순 IT 플랫폼 주장만으로는 운송 알선 시 무등록 여행업 리스크 발생 가능
- 중개 전환 시 장부상 매출이 급변(GMV vs 수수료) → 세무/투자 유치 평가 지표(Revenue)와 괴리
- 이 단계는 현재 로드맵 1-4단계 완주 전에는 착수하지 않음
