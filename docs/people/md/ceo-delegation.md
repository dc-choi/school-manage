# 대표 위임 체크리스트

향후 채용 또는 외부 협업이 확정됐을 때 기술과 콘텐츠 업무를 안전하게 넘기기 위한 템플릿이다.
현재 인력이 채용됐거나 위임이 시작됐다는 뜻이 아니다. 2026-07-15 감사 기준 상태이며,
현재 운영 주체와 사업 상태는 `docs/business/STATUS.md`, 역할별 실행 절차는 이 디렉토리의
온보딩 문서를 따른다.
후보 판단과 합류 전 핸드오프는 `role-profiles.md`의 역할별 기준과 기록 양식을 사용한다.

## 시작 전 게이트

| 역할 | 시작 조건 | 현재 문서상 상태 |
| --- | --- | --- |
| 프로덕트 엔지니어 | 담당자, 계약 범위, 첫 백로그, 접근 승인자 확정 | 채용 또는 위임 확정 기록 없음 |
| 콘텐츠 마케터 | 담당자, 계약 범위, 채널 권한 확정 | Instagram 재개가 STATUS에 별도 승인되기 전까지 콘텐츠 생산 동결 |

- 담당자와 시작일을 이 문서에 직접 넣지 않는다. 계약 또는 인사 기록에서 확인한 뒤 온보딩 티켓에 기록한다.
- 인스타 재개 여부는 `docs/business/3_gtm/instagram.md`의 게이트를 같은 기준일에 평가하고
  `STATUS.md`에 결정이 기록된 뒤 확정한다.
- 현재 수치, 가격, 기능 완료 여부를 복제하지 않는다. 각각 `STATUS.md`,
  `docs/business/7_pricing/pricing.md`, `docs/specs/README.md`와 코드를 확인한다.

- [ ] 배포 절차, 장애 대응, 롤백 방법을 현재 워크플로와 대조
- [ ] 초기에는 운영 DB 읽기 전용 접근만 준비

### 콘텐츠 마케터

- [ ] `docs/business/3_gtm/instagram.md`에서 동결 또는 재개 상태 확인
- [ ] 현재 공개 유지 피드와 `docs/content/README.md`의 전체 이력 구분
- [ ] 사용 승인된 로고, 캐릭터, 스크린샷 확인
- [ ] Insights 기준선은 조회일, 기간, 지표 단위를 함께 기록
- [ ] 동결 중에는 신규 콘텐츠와 게시 일정을 배정하지 않고 기존 콘텐츠 검수만 수행

### 공통

- [ ] 선발 근거, 확인된 강점, 학습 공백, 첫 산출물을 한 장으로 기록
- [ ] 프로덕트 데모와 테스트용 계정 준비
- [ ] `docs/business/README.md`, `STATUS.md`, `docs/specs/README.md` 읽기
- [ ] AI 가이드 3종 전달: 설치, 기초, 실전 순서
- [ ] 업무용 개별 계정, MFA, 권한 회수 담당자 지정
- [ ] 비밀값, 개인정보, 비공개 운영 원본을 개인 기기나 AI 대화에 입력하지 않는 원칙 확인

## Day 1 합동 온보딩

| 세션 | 내용 | 권장 시간 |
| --- | --- | ---: |
| 프로덕트 데모 | 가입, 조직, 그룹, 학생, 출석, 대시보드 흐름 | 20분 |
| 사용자와 포지셔닝 | 현재 PMF 타깃과 외부 메시지, 확장 비전 비공개 원칙 | 10분 |
| 사업 상태 | `STATUS.md`의 기준일, MAO와 WVO 등 지표 단위 | 10분 |
| 업무 도구 | 역할별 안전한 읽기 작업과 승인 흐름 시연 | 20분 |

숫자는 슬라이드에 고정하지 않고 세션 당일 `STATUS.md`에서 읽는다. MAO 같은 단체 지표와
MAU 같은 사용자 지표를 하나의 전환 퍼널로 섞지 않는다.

## 4주 위임 검증

4주는 자동 완료 기한이 아니라 역량과 통제 장치를 확인하는 기본 검증 기간이다.

| 주차 | 프로덕트 엔지니어 | 콘텐츠 마케터 | 대표 역할 |
| --- | --- | --- | --- |
| 1주 | 로컬 환경, 테스트, 배포 흐름 관찰 | 제품, 사업 문서, 기존 콘텐츠 리뷰 | 데모, 접근 승인, 질문 대응 |
| 2주 | 낮은 위험의 첫 PR과 CI 확인 | 동결 중 기존 콘텐츠 검수, 재개 후 첫 게시물 리뷰 | 첫 산출물 승인 |
| 3주 | 명시된 범위의 독립 구현과 운영 연습 | 승인 상태에 맞춘 초안 또는 채널 운영 | 비동기 리뷰와 예외 판단 |
| 4주 | 배포, 장애 대응, 롤백 리허설 | DM, 게시, 지표 보고 인수 검증 | 인수 승인 또는 기간 연장 |

- 콘텐츠 동결 중에는 2~4주의 제작과 게시 목표를 기존 콘텐츠 검수와 DM 시나리오 연습으로 대체한다.
- 프로덕션 배포와 실제 게시 권한은 주차가 아니라 수용 기준 충족 여부로 연다.
- 수용 기준을 못 채우면 범위를 줄이거나 검증 기간을 연장한다.

## 최소 권한 이양

공용 비밀번호를 문서나 메신저로 전달하지 않는다. 가능한 경우 개별 사용자와 역할 기반 권한을
사용하고, 불가피한 공용 자격 증명은 승인된 비밀번호 관리자에서만 공유한다.

### 프로덕트 엔지니어

| 접근 | 초기 권한 | 확대 조건 |
| --- | --- | --- |
| GitHub | 필요한 저장소 write | 브랜치 보호와 리뷰 규칙 확인 |
| GitHub Actions | 실행과 로그 조회 | secrets 관리 권한은 별도 승인 |
| AWS Lightsail | 조회 또는 제한된 운영 역할 | 배포와 롤백 리허설 통과 |
| Docker Hub | 저장소 단위 권한 | 이미지 배포 절차 확인 |
| 운영 DB | 읽기 전용 | 쓰기와 마이그레이션은 건별 승인 |

### 콘텐츠 마케터

| 접근 | 초기 권한 | 제한 |
| --- | --- | --- |
| Instagram | Insights와 수신 DM 운영 | 게이트 충족 전 신규 게시 금지 |
| Google Form | 응답 확인 | 개인정보 최소 조회, 외부 반출 금지 |
| GA4 | Viewer | 사용자 단위 원본 내보내기 금지 |
| 제작 도구 | 업무 범위 계정 | 학생, 교사 개인정보 입력 금지 |

## 의사결정과 에스컬레이션

| 상황 | 1차 담당 | 대표 승인 또는 보고 |
| --- | --- | --- |
| 프로덕션 장애 | 엔지니어 | 즉시 보고, 데이터 변경과 롤백 승인 |
| 사용법 문의와 수신 DM | 마케터 | 약속, 환불, 법률, 개인정보 사안 이관 |
| 기술 버그 제보 | 마케터가 재현 정보 수집 | 엔지니어에게 이관 |
| 사제, 교구, 파트너 관계 | 대표 | 대표가 직접 담당 |
| 가격, 사업 전략, 기능 우선순위 | 대표 | 각 사업 단일 출처에 결정 반영 |
| 개인정보 요청 또는 사고 | 대표와 엔지니어 | 즉시 접근 차단과 대응 절차 실행 |

## 4주 후 인수 판정

### 대표가 유지하는 결정권

- 사업 전략, 가격, 계약, 파트너십
- 사제와 교구 관계
- 개인정보, 법률, 재무 리스크 승인
- 기능 우선순위와 프로덕션 데이터 변경 승인
- 인스타 재개, 중단, 예산 결정

### 수용 기준

| 영역 | 확인 방법 |
| --- | --- |
| 엔지니어 독립성 | 지정 이슈를 테스트, PR, 배포 검증까지 수행하고 롤백 절차 설명 |
| 마케터 독립성 | 승인된 운영 상태 안에서 초안 또는 게시, DM, 지표 보고 수행 |
| 문서화 | 반복 업무와 예외 처리 절차가 저장소 또는 승인된 운영 문서에 남음 |
| 보안 | 최소 권한, MFA, 비밀값 비노출, 개인정보 비반출 확인 |
| 서비스 연속성 | 담당자 부재 시 대체 연락망과 권한 회수 절차 확인 |
