# PRD: 학생 검색 개선

> 상태: Draft | 작성일: 2026-03-20

## 배경/문제 요약

- 참고: `docs/business/STATUS.md`, `docs/business/0_feedback/feedback.md`
- 문제: 현재 학생 검색은 단일 필드 선택 후 버튼 클릭 방식으로, 직관적이지 않고 빠른 조회가 어려움
- 현재 상태:
  - 검색 옵션(이름/세례명/축일) 중 하나를 선택한 뒤 검색어를 입력하고 버튼을 클릭해야 함
  - "전체" 옵션은 실제로 검색하지 않음 (필터링 없이 전체 목록 반환)
  - 실시간 검색 없음 (버튼 클릭 필수)
  - 한 번에 하나의 필드만 검색 가능
- 목표 상태: 검색어 입력 시 이름/세례명/축일을 동시에 검색하는 통합 검색, 실시간 반영

## 목표/성공 기준

- **목표**: 학생 검색 UX를 개선하여 "빠른 조회" 핵심 가치 강화
- **성공 지표**: 검색 사용률 증가 (GA4 이벤트 기준)
- **측정 기간**: 출시 후 4주

## 사용자/대상

- **주요 사용자**: 교리교사 (ADMIN, TEACHER)
- **사용 맥락**: 매주 일요일 미사/교리 시간 전후, 학생을 빠르게 찾아야 하는 상황 (출석 체크, 학생 정보 확인, 연락처 조회)

## 범위

### 포함

- 통합 검색: 검색어 하나로 이름/세례명/축일을 동시에 검색
- 실시간 검색: 입력 즉시 결과 반영 (디바운스 적용)
- 검색 UX 개선: 검색 옵션 드롭다운 제거, 단일 입력 필드로 통합

### 제외

- 퍼지 매칭 / 자동완성
- 검색 히스토리
- 추가 학생 필드 (부모님 연락처, 생년월일 등) — 별도 PRD로 분리

## 사용자 시나리오

1. **빠른 이름 검색**: 교리교사가 학생 목록에서 검색창에 "김"을 입력하면, 이름에 "김"이 포함된 학생 + 세례명에 "김"이 포함된 학생이 즉시 표시됨
2. **축일 검색**: 검색창에 "03"을 입력하면, 축일(baptizedAt)에 "03"이 포함된 학생이 표시됨
3. **검색 초기화**: 검색창을 비우면 전체 학생 목록으로 복귀

## 요구사항

### 필수 (Must)

- [ ] 통합 검색: 하나의 검색어로 societyName, catholicName, baptizedAt을 OR 조건으로 동시 검색
- [ ] 실시간 검색: 디바운스(300ms) 적용, 입력 즉시 결과 반영
- [ ] 검색 옵션 드롭다운 제거: 단일 입력 필드로 통합
- [ ] 검색 시 페이지 1로 리셋
- [ ] 기존 필터(삭제/졸업/등록)와 검색 조합 유지

### 선택 (Should)

- [ ] 검색 결과 수 표시 (예: "3명 검색됨")
- [ ] 검색 활성 상태 시각적 표시 (검색 중 배지/아이콘)
- [ ] 검색어 클리어 버튼 (X 아이콘)

### 제외 (Out)

- 퍼지 매칭, 자동완성, 검색 히스토리
- 검색 인덱스 추가 (소규모 데이터, 테이블 스캔 충분)
- 서버 사이드 full-text search 도입

## 제약/가정/리스크/의존성

- **제약**: 검색은 accountId 소속 학생 범위 내에서만 수행 (기존 권한 모델 유지)
- **가정**: 학생 수가 소규모(~200명/모임)이므로 DB substring 검색으로 성능 충분
- **리스크**: OR 조건 검색 시 의도하지 않은 결과 포함 가능 (예: 숫자 검색 시 이름+축일 동시 매칭)
- **내부 의존성**: `@school/shared` 스키마 변경, `@school/trpc` 프로시저 수정
- **외부 의존성**: 없음

## 롤아웃/검증

- **출시 단계**: 전체 사용자 동시 적용 (기존 검색 대체)
- **이벤트**: `student_search` (검색 실행 시) | **검증**: GA4에서 검색 사용률 및 검색어 패턴 확인
