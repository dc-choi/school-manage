# 코딩 스타일 (공통)

`shared.md`는 `@school/shared` 패키지 가이드이고, 본 문서는 **모든 패키지에 공통 적용되는 일반 코딩 표준**이다. 언어 특화 규칙은 `typescript.md` 참조.

## 불변성 (CRITICAL)

원본 객체를 변경하지 말고 항상 새 객체를 반환한다.

```typescript
// 잘못됨
function setName(user, name) { user.name = name; return user; }

// 올바름
function setName(user, name) { return { ...user, name }; }
```

이유: 숨은 사이드 이펙트 방지, 디버깅 용이, 동시성 안전.

## 핵심 원칙

### KISS (Keep It Simple)
- 실제로 동작하는 가장 단순한 해법을 우선
- 조기 최적화 금지
- 영리함보다 명확함

### DRY (Don't Repeat Yourself)
- 반복 로직은 공유 함수/유틸로 추출
- 복붙 후 미세 수정으로 분기 만들지 말 것
- **추측이 아니라 실제 반복이 발견됐을 때만** 추상화

### YAGNI (You Aren't Gonna Need It)
- 필요해지기 전까지 기능/추상화를 만들지 않는다
- 사변적 일반화 금지
- 단순하게 시작하고 압력이 실제로 발생했을 때 리팩터

## 파일 조직

작은 파일 다수 > 큰 파일 소수:
- 응집도 높이고 결합도 낮춤
- 200~400줄 권장, **800줄 상한**
- 800줄 넘으면 모듈로 추출
- 폴더는 도메인/기능 단위로 (타입별 X)

## 에러 처리

- 모든 레이어에서 명시적으로 처리
- UI 노출 코드는 사용자 친화적 메시지 제공
- 서버는 상세 컨텍스트 로깅
- **에러를 조용히 삼키지 말 것** (silent-failure-hunter 에이전트 참고)

## 입력 검증

시스템 경계에서 항상 검증한다:
- 사용자 입력은 처리 전 반드시 검증 (Zod 권장)
- 외부 데이터(API 응답, 사용자 입력, 파일 내용)는 신뢰하지 않음
- 빠르게 실패하고 명확한 메시지 제공

## 명명 규약

- 변수/함수: `camelCase`, 서술적 이름
- Boolean: `is`, `has`, `should`, `can` 접두
- 인터페이스/타입/컴포넌트: `PascalCase`
- 상수: `UPPER_SNAKE_CASE`
- 커스텀 훅: `useXxx`

## 피해야 할 코드 냄새

### 깊은 중첩
중첩이 쌓이기 시작하면 early return 우선.

### 매직 넘버
임계값/지연/한계는 명명된 상수로.

### 긴 함수
큰 함수는 책임이 명확한 작은 조각으로 분리.

## 완료 전 체크리스트

작업을 완료로 표시하기 전:
- [ ] 코드가 잘 명명되어 읽기 쉬움
- [ ] 함수가 작음 (**<50줄**)
- [ ] 파일이 집중됨 (**<800줄**)
- [ ] 깊은 중첩 없음 (**<4단계**)
- [ ] 에러 처리 적절
- [ ] 하드코딩된 값 없음 (상수/설정 사용)
- [ ] 변경(mutation) 없음 (불변 패턴 사용)
