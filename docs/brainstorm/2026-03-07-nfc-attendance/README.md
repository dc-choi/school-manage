# NFC 출석 기술 리서치

## 주제

NFC 기반 출석 체크 시스템 — 외부 NFC 리더기가 HTTP 요청으로 출석 기록 + 서버 TTS 음성 피드백

## 배경

- 로드맵 2단계: "NFC 출석 유료 파일럿" / 3단계: "NFC 출석 유료 애드온 정식 출시"
- 현재 출석 방식: 교사가 웹앱에서 수동 입력 (체크박스)
- 목표: NFC 카드 태깅으로 출석 자동화, 교사 행정 부담 감소

## 기술 옵션 검토

### 옵션 1: Web NFC API (웹앱에서 직접)

- Chrome Android 89+에서만 지원
- **iOS 미지원** (Safari, Chrome 모두 불가)
- 한국 아이폰 점유율 ~35% → 교리교사 중 아이폰 사용자 이용 불가
- **기각**: 플랫폼 제약이 심각

### 옵션 2: 네이티브 앱 (React Native/Flutter)

- iOS/Android 모두 NFC 지원 가능
- 별도 앱 개발 + 앱스토어 배포 + 유지보수 필요
- **보류**: 1인 개발 체제에서 비현실적

### 옵션 3: QR 코드 대안

- iOS/Android 모두 가능, 개발 비용 낮음
- NFC 대비 1~2초 느림 (카메라 앱 열기 필요)
- **대안으로 유지**: NFC보다 접근성 높음, 별도 하드웨어 불필요

### 옵션 4: 외부 NFC 리더기 + HTTP API (**채택**)

- NFC 리더기가 독립 디바이스로 동작, WiFi를 통해 API 서버에 HTTP 요청
- iOS/Android 제약 없음 (브라우저 의존 X)
- 하드웨어 비용 저렴 (본당당 ~35,000원부터)

## 채택 아키텍처

### 전체 흐름

```
[학생 NFC 카드]
     ↓ tap
[NFC 리더기 (ESP32 또는 라즈베리파이)]
     ↓ HTTP POST (tagId)
[출석부 API 서버 (Node.js/tRPC)]
     ├→ 출석 기록 (DB)
     ├→ TTS 생성 ("홍길동, 출석 완료! 올해 34번째!")
     └→ 응답: { studentName, message, audioUrl }
            ↓
[리더기 → 스피커] → audioUrl 스트리밍 재생
```

### 버전 로드맵

| 버전 | 플랫폼 | 기능 | 대상 |
|------|--------|------|------|
| v1 기본형 | ESP32 + PN532 + 스피커 | NFC 태깅 → 출석 기록 → TTS 음성 피드백 | 모든 본당 |
| v2 프리미엄 | 라즈베리파이 + PN532 + 스피커 + 모니터 | v1 + 출석 현황 화면 시각화 | 시각화 니즈 있는 본당 |

### 버전별 상세 문서

- **v1**: `docs/brainstorm/2026-03-07-nfc-attendance/v1.md`
- **v2**: `docs/brainstorm/2026-03-07-nfc-attendance/v2.md`

### 본당 운영 비용 비교

| 항목 | v1 (ESP32) | v2 (라즈베리파이) |
|------|-----------|---------------|
| 리더기 1대 | ~35,000원 | ~260,000~390,000원 |
| 학생 카드 (40명) | ~12,000원 | ~12,000원 |
| **본당당 합계** | **~47,000원** | **~272,000~402,000원** |
| 월 유지비 | 0원 | 0원 (전기세 무시 수준) |

### NFC 카드 (v1/v2 공통)

- **모델**: NTAG215 PVC 카드
- **형태**: 교통카드 크기 또는 스티커형
- **메모리**: 504바이트, 고유 UID 보유
- **읽기 방식**: 리더기에 밀착~3cm 터치 (교통카드와 동일)
- **가격**: 대량 구매 시 ~250~400원/개
- **구매처**: G마켓, 쿠팡, 알리익스프레스

## 백엔드 변경 사항

### DB 변경

- Student 테이블에 `nfc_tag_id` 필드 추가 (NFC 카드 UID, nullable)

### 새 tRPC 프로시저

| 프로시저 | 타입 | 설명 |
|----------|------|------|
| `nfc.checkIn` | mutation | 태그 UID → 출석 기록 + 학생 정보 + TTS 오디오 URL 반환 |
| `nfc.registerTag` | mutation | 태그 UID ↔ 학생 매핑 |
| `nfc.unregisterTag` | mutation | 태그 매핑 해제 |

### TTS 처리

- 서버에서 TTS 생성 (네이버 Clova / Google Cloud TTS 등)
- 응답에 audioUrl 포함 → 리더기가 스트리밍 재생
- 대안: 서버에서 MP3 생성 후 응답 body에 바이너리 포함

## 보안 고려사항

| 위협 | 대응 |
|------|------|
| 리더기 위장 | API 키 + deviceId 인증 |
| 카드 복제 | 동일 UID 동일 날짜 중복 출석 방지 (서버 측) |
| 중복 태깅 | 같은 카드 동일 날짜 1회만 유효 |
| 네트워크 차단 | 로컬 버퍼링 → WiFi 복구 시 일괄 전송 |

## 웹앱 변경 사항

- 학생 관리: NFC 카드 등록/해제 UI
- 출석 페이지: NFC 출석과 수동 출석 통합 표시
- 출석 로그: NFC 출석 시간/디바이스 정보 조회

## 리스크

| 리스크 | 심각도 | 대응 |
|--------|--------|------|
| ESP32 펌웨어 개발 경험 없음 | 높음 | 오픈소스 예제 활용, AI 코딩 지원 |
| 본당 WiFi 환경 불안정 | 중간 | 로컬 버퍼링, 핫스팟 대안 |
| NFC 카드 분실/파손 | 낮음 | 재발급 300원, 수동 출석 병행 |
| 리더기 고장 | 낮음 | 수동 출석으로 폴백 |
| TTS API 비용 | 낮음 | 소량이면 무료 티어 가능 |

## 의존성

- 계정 모델 전환 (2단계 선행 과제) 완료 후 진행이 효율적
- NFC 태그 ↔ 학생 매핑은 계정 모델과 무관하게 독립 구현 가능

## 결론

- 외부 NFC 리더기 + HTTP API 방식이 비용/호환성/확장성 면에서 최적
- 서버 TTS로 동적 음성 피드백 제공
- MVP는 ESP32(v1), 시각화 니즈 시 라즈베리파이(v2)로 업그레이드
- SDD 착수 전 하드웨어 프로토타입 검증 권장
