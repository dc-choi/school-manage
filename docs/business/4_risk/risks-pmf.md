# 리스크 — 사용자/도입 (PMF)

`risks.md` 분리 — 도입 저항/첫인상/정착/시험기간/Silent Churn/도입 후 활성화/보안 인식 등 PMF 단계 리스크. 인덱스: `risks.md`.

## 리스크 1: 보수적 문화로 인한 도입 저항

- 가설: 가톨릭 조직 문화 특성상 새로운 디지털 도구 도입이 느리거나 거부될 수 있다
- 근거: "가톨릭은 보수적"이라는 인식, 기존 방식(수기/엑셀) 유지 경향. 봉헌금 페이 서비스 상용화 실패 사례 (2026-01-25 피드백)
- 검증: 도입 제안→승인 전환율, 보류/거절 기록
- 대응: 소규모 파일럿과 성공사례 공유로 신뢰 기반 추천 유도

## 리스크 2: 사제 동의 지연

- 가설: 최종 의사결정자인 사제의 동의가 지연되면 파일럿 이후 전환이 늦어진다. **본당 차원 = 본당 사제(주임신부) / 교구 차원(양업시스템 연동/교구 예산) = 주교** (2026-05-09 위계 분리)
- 근거: 도입과 결제의 최종 승인이 사제에게 있음. **사제 네트워크 시스템화 어려움 시그널 (2026-05-07 갱신)** — 03-14 +57 계정 추천 단일 사례 후 53일 단절. 5/6 후속 사적 미팅에서 어젠다 4건(보고+감사/Footer 공유/추천/PG 청취) **모두 미진입**(사적 분위기상 부적절), 응원만 수령. 자연 발생 의존 채널, 시스템화 가설 약화
- **주교 접점 게이트 (2026-05-09 신규)**: 양업시스템 연동/교구 차원 도입은 청소년국/교리교사 추천만으로는 격상 트리거 발화 X — **주교 접점 확보가 진짜 게이트**. 2026-05-08 카톡 토론에서 외부 추천(이한별)으로 양업 격상 제안 받았으나, 청소년국은 협력 채널일 뿐 의사결정 권한 없음 확인. 닭/달걀 인식(바이럴 → 양업 호출) 부분 유효 + 주교 접점이 본질. **주교 접점 자연 발생 채널 = 본당 사제 네트워크 → 사제 간 추천 → 교구 사제 회의 노출 → 주교 인지** (다단 자연 트리거)
- 검증: 파일럿 종료 후 동의까지 리드타임, 보류/거절 코드. **사제 네트워크 자연 발생 추천 빈도** (단일 사례 후 N일 단절 추세). **주교 인지 도달 여부** (사제 네트워크 다단 노출 후 주교급 응답 발생 시점 측정)
- 대응: 사제 보고용 요약(효과/통계) 템플릿 제공으로 동의 리드타임 단축. **시스템화 X — 자연스러운 접촉 대기 (영업 미팅 X 원칙 유지)**. **주교 접점 사전 준비물**: R25 보안 신뢰 시그널 + 파일럿 1페이지 자료(MAO/본당 수/출석 건수) — 자연 트리거 발생 시 즉시 활용 가능하도록 미리 작성

## 리스크 3: 첫 경험 실패 후 장기 미사용

- 가설: 첫 경험에서 가치를 느끼지 못하면 재시도 없이 장기 미사용으로 이어질 수 있다. 30일 미활동만으로 영구 이탈을 확정하지 않는다
- 근거: 고덕동은 그룹 1개 생성 후 미사용이 지속됐다. 2026-02-18 발송문은 사용법 안내가 아니라 일반적인 불편사항 질문이었고, 답변은 발송 시점까지 제대로 사용하지 않았음을 보여준다 (`0_feedback/entries/2026-02-18-godeokdong.md`)
- 증거 한계: 이 사례는 미사용 지속과 낮은 피드백 의향의 신호지만, DM이 침묵을 유발했다거나 구체적 온보딩 안내가 효과 없음을 입증하지 않는다. D유형 맞춤 안내 실행을 폐기한 운영 결정과 가설의 검증 상태를 분리하며, 해당 가설은 **미검증**으로 둔다
- 검증: 첫 세션 이탈률과 가입 후 첫 출석 기록까지 도달률. 현재 사용자 이벤트와 단체 이벤트가 섞여 있어 유효 전환율은 미측정이며, 측정 가능 여부와 최신값은 `STATUS.md`의 지표 유효성을 단일 출처로 사용한다
- 대응: 온보딩 각 단계에서 즉각적 결과를 제공하고, 사람 DM/구체적 사용 안내/시스템 알림을 서로 다른 개입으로 측정한다

## 리스크 5: B2B2C 구조로 인한 최종 사용자 정착 실패

- 가설: 도입 결정 주체와 실제 사용자가 분리되어 있으면 최종 사용자(교리교사) 정착이 어려울 수 있다
- 근거: 사제(결정)와 교리교사(사용)의 역할 분리 + 본당 계정 1개 공유 사례(성북동 미카엘라 04-19)에서 입력자/조회자 역할이 갈라짐 — 도입 동기와 실사용 동기가 분리되는 패턴 관찰
- 검증: 도입 경로별 4주 리텐션 및 활성도 비교
- 대응: 최종 사용자의 즉시 효용(출석/학생 관리) 강조

## 리스크 12: 가치 체감 부족으로 인한 빠른 이탈

- 가설: 진입 장벽을 낮춰도 핵심 가치가 체감되지 않으면 빠르게 이탈한다
- 근거: 피드백 이벤트 응답 2건은 "행동 비용 대비 보상이 불확실하면 행동하지 않는다"는 가설의 약한 신호다. 최신 R24-A0/A1 코호트와 추가 표본으로 검증해야 한다 (`docs/business/1_market/market.md` 행동-보상 원칙)
- 검증: 초기 핵심 기능 사용률과 2주 이탈률 상관
- 대응: 최소 입력 플로우 + 입력 즉시 결과 제공 ("행동 비용은 낮게, 보상 확실성은 높게")

## 리스크 18: 본업(생업/학업) 이슈로 인한 도구 전환/사용 시간 부족

- 가설: 교리교사는 봉사직이므로 생업/학업/시즌에 따라 사용 공백과 복귀가 반복될 수 있다. 입력 공백만으로 실제 운영 중단이나 영구 이탈을 단정하지 않는다
- 근거: 과거 `STATUS.md` 이력에는 14-56일 뒤 복귀와 복귀 후 재이탈이 기록돼 있다. 특히 56일 뒤 복귀 사례로 28-30일을 영구 이탈 경계로 보던 가설은 반증됐다. 최신 사례 수는 새 운영 쿼리로 다시 산출하고, 현재 판정 규칙은 `5_metrics/metrics.md`를 따른다
- 운영상 구분:
    - 14일 미만: 최근 활동
    - 14-28일: 휴면 관찰 구간. 시험/명절/시즌 또는 입력 지연 가능성을 함께 본다
    - 28-30일: MAO 30일 창 이탈 임박 구간일 뿐 영구 이탈 판정이 아니다
    - 30일 이상: MAO 창 이탈. 영구 이탈은 명시적 거부/탈퇴 또는 별도 합의한 장기 관찰 기준으로만 확정한다
- 측정 한계: 출석 발생일과 입력일이 다른 사후 일괄 패턴이 있으므로 `attendance.date`와 `createdAt`을 분리하지 않은 미입력 일수는 이탈 판정 근거로 사용할 수 없다
- 검증: 휴면 후 복귀율, 복귀 후 재이탈률, 시즌별 MAO 변동, 입력 지연 일수 분포를 함께 측정한다. 7-8월 방학 효과와 9월 복귀율도 같은 기준으로 본다
- 대응: 최소 입력 온보딩과 컨텍스트 배너로 행동 비용을 낮춘다. 비활성 알림을 재구현할 경우 R24-A0/A1과 입력 지연 후보를 분리하고, 일수만으로 "영구" 메시지를 발송하지 않는다

## 리스크 19: Silent Churn으로 인한 문제 파악 지연

- 가설: 2030세대는 불만이 있어도 피드백 없이 조용히 이탈하여, 원인 파악이 지연된다
- 근거: 피드백 이벤트 응답이 2건에 그쳤고, 고덕동은 구체적인 불편 원인을 밝히지 않은 채 미사용이 지속됐다
- 증거 한계: 고덕동 DM은 이미 미사용 중인 운영자에게 보낸 일반 피드백 요청이다. DM이 침묵을 유발했다는 인과나 구체적 온보딩 안내의 효과를 검증하지 않았다. 2026-05-19 능동 DM 실행 폐기 결정은 운영 결정으로 보존하되, 가설 반증으로 표기하지 않는다 (`0_feedback/entries/2026-02-18-godeokdong.md`)
- 검증: 피드백 수집률, 이탈 후 원인 파악까지 소요 시간
- 측정 한계: 입력 지연 후보는 미입력 기간이 실제 운영 중단과 다를 수 있으므로 R24 분류를 먼저 적용한다
- 구현 상태: GA4 행동 이벤트 추적은 운영 중이다. 14일 미활동 운영자 이메일은 **미구현 상태**다. 2026-05-07 마이그레이션에서 09:00 이탈 감지 cron과 `churn_alert_log`가 제거됐다 (`apps/api/prisma/migrations/20260507/drop_churn_alert_log.sql`). R24-A0 학생 0 알림도 별도 오픈 이슈이며 기존 감지 시스템으로 간주하지 않는다
- 대응: 알림 재구현 전 R24-A0/A1과 입력 지연 확정/후보를 분리한다. 발송 후 7일 내 첫 학생/첫 출석/복귀율을 측정해 회수 가치를 검증한다

## 리스크 24: 도입 후 활성화/초기 정착 실패

- 가설: 모임 생성 뒤 첫 학생/첫 출석에 도달하지 못하는 문제와, 첫 출석 뒤 사용이 이어지지 않는 문제는 서로 다른 활성화 실패다
- 분류 단일 출처: 단체별 최신 분류/수치/후보 목록은 `STATUS.md`의 파일럿과 리스크 판정을 단일 출처로 사용한다. 본 문서는 정의와 판정 규칙만 유지한다. 기존 ①/③/④ 숫자 라벨은 과거 분석 이력이며 신규 실험군 이름으로 사용하지 않는다
- 분류 규칙:
    - **R24-A0 미활성화**: 모임은 생성했지만 학생 0 또는 첫 출석 0. 학생/출석 데이터로 직접 확인되면 확정한다
    - **R24-A1 활성화 후 휴면**: 첫 출석은 기록했지만 후속 사용이 이어지지 않음. 입력 지연을 배제하기 전에는 후보로 둔다
    - **입력 지연 행동 축**: 출석은 진행하지만 입력이 늦는 패턴으로, A0/A1과 별도 축이다. `attendance.date`와 `createdAt` 검증으로 확인한 단체만 확정하고, 요일/총량 변화만으로 추정한 단체는 후보로 둔다
- 실험 사용 규칙: 확정 코호트만 전환율/WTP/리텐션 분모에 넣는다. 후보와 혼합 분류는 별도 보류군으로 두고, 입력 지연 확정/후보 태그를 교차 적용한다. `STATUS.md`가 갱신되면 연결 문서도 같은 분류를 사용한다
- 검증: A0는 모임 생성 후 7일 내 첫 학생 등록률과 14일 내 첫 출석률, A1은 첫 출석 후 2회차 출석률과 복귀율, 입력 지연 축은 지연 일수와 사후 일괄 비율을 측정한다
- 대응:
    - A0: 3일 학생 0 운영자 이메일, 7일 학생 0 가입자 인앱/이메일을 검증한다. 현재는 오픈 이슈이며 구현 완료가 아니다
    - A1: 입력 지연과 R18 휴면을 배제한 뒤 별도 복귀 알림을 검증한다
    - 입력 지연 축: 비활성 알림 분모에서 후보/확정 상태를 구분하고, 확정 단체에는 입력 패턴에 맞춘 임계값을 적용한다

## 리스크 25: 보안 인식 기반 도입 거부 (R15와 분리)

- 가설: 서버 기반 SaaS의 "데이터 소유권 부재" 인식과 "해킹 시 개인정보 유출 책임" 우려가 도입 게이트키퍼(교구 청소년국 직원, 사제, 학부모)에게 전파되면, 법적 컴플라이언스(R15) 충족 여부와 무관하게 도입 자체를 거부할 수 있다
- **R15와의 차이**: R15는 법적 미준수에 따른 규제/과태료 리스크(공급자 책임), R25는 보안 인식 기반 도입 거부 리스크(수요자 인식). 동의 절차/법률 자문이 완료되어도 R25는 신뢰 시그널 부재 시 잔존 (R15: `risks-business.md`)
- 근거:
    - **2026-05-02 서울대교구 청소년국 중고등부 간사 피드백** — "서버 기반이라 데이터 소유권 우려 + 해킹 시 개인정보 유출 책임" 이유로 "사용을 꺼린다" 워딩 (대면 대화, 개인 차원 의견). 도입 게이트키퍼급 페르소나 첫 보안 인식 시그널 (`0_feedback/entries/2026-05-02-cheongsonyeon-ganji.md`)
    - **보안 미측정 도입 사례**: 2026-03-27 군종교구 목포해군성당은 자발적 도입 후 학생 수정 버그를 제보했다 (`0_feedback/entries/2026-03-27-gunjong.md`). 원문에는 보안/데이터 소유권 질문이나 승인 절차가 없어 R25 게이트 통과 또는 반례로 사용할 수 없다. 직접 운영자와 게이트키퍼의 인식 차이는 추가 인터뷰 전까지 가설이다
    - 가톨릭 보수적 문화(R1) + 학생/보호자 정보 민감성(R6→R15 통합) 누적 → 보안 인식 임계점 낮을 가능성
