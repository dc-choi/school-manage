# 브레인스토밍 결과 보고서

## 주제
GTM(Go-To-Market) 전략 재수립 — 인스타그램 4컷 만화 가시성 저조, 파일럿 16곳 중 사용 중 3곳, WAU 7명 상황에서 효과적인 사용자 획득 및 확산 전략

## 토론 요약

### Round 1
- 기획자 3개 아이디어 제안: (1) 4성북지구 집중 공략 — 지구 단위 도미노 확산, (2) 가입 후 미사용자 맞춤 팔로업, (3) 인스타 콘텐츠 피벗 — 교리교사 공감 콘텐츠
- 비판자 핵심 지적: 세 전략 모두 계정 모델 전환이 선행되어야 함. 전략 2의 "3분이면 됩니다" 단일 메시지는 이탈 원인이 다른 6곳에 일률 적용 불가. 전략 1은 길음동 안착이 먼저. 전략 3은 공감→가입 연결 고리가 약함
- 합의: A/B/C 유형 분류 도입, 교리교사 수평 추천 실증 확인, 인스타 착지점 재정의

### Round 2
- 기획자 개선: 팔로업 원인별 3유형 분류(A: 바빠서, B: 구조 혼란, C: 가치 체감 부족), 길음동 안착 2단계 분리, 2트랙 병행(개발+GTM) 구조 제시
- 비판자 핵심 지적: 유형 분류가 추정에 기반(진단 아님), 2트랙 시간 비용 미계산, 마이그레이션 가속 위험(잘못된 구조로 데이터 축적), 성남동 추가 필드 시그널 무시
- 합의: 25h/주 시간 예산, 마이그레이션 우선순위 차등화, 길음동 안착 기준 2단계

### Round 3
- 기획자 최종 개선: WAU 목표 15→12명 하향, IDOR 핫픽스 3월 즉시 처리로 격상, 피드백 트리거 4개 조건 설계, 마이그레이션 공지 3항목 구조, 돈암동 B유형 확정
- 비판자 최종 지적: 2월 4주 액션 집중 분산 필요, SDD 3월 말 완료 일정 도전적, 피드백 결과가 로드맵 뒤집을 분기점
- 최종 합의 13개 항목 확정, 미합의 0개 (전부 수용 또는 조건부 합의)

## 아이디어 목록

### [아이디어 1 → 전략 C] 파일럿 확산 — 유형별 안착 전환

- 한 줄 요약: 자가 가입한 파일럿 본당들을 원인별 유형 분류 후 맞춤 대응하여 안착으로 전환
- 원래 형태: 4성북지구 도미노 확산 + 미사용자 팔로업 (전략 1+2 통합)
- 사용자 가치: 가입 후 "뭘 해야 할지 모르는" 사용자를 안착까지 안내
- 실행 방법:
  - 유형 A(바빠서): "멤버만 넣어두세요" 메시지, 즉시 실행
  - 유형 B(구조 혼란): 계정 모델 전환 후 재접촉
  - 유형 C(가치 체감 부족): 30분 통화로 직접 온보딩, C(추정) → 통화 후 확정
- 차별점: 범용 앱은 아무도 팔로업하지 않음. 1인 개발자의 직접 온보딩이 차별점
- 판정: **승인**

### [아이디어 2 → 전략 A] 계정 모델 전환 (공유 → 개인 + 본당 합류)

- 한 줄 요약: 공유 계정 구조를 개인 계정 + 본당/모임 합류 구조로 전환하여 구조적 이탈 원인 제거
- 원래 형태: 3개 전략의 공통 선행 조건으로 도출됨 (기존 전략에는 없던 것)
- 사용자 가치: 신당동/금호동의 "여러 명 개별 가입 혼란" 해소
- 실행 방법: SDD 설계(3월) → 구현(4월) → 마이그레이션(장위동 우선)
- 차별점: 구조적 문제 해결 — UX 개선이 아닌 아키텍처 전환
- 판정: **승인 (최우선)**

### [아이디어 3 → 전략 B] 피드백 기반 로드맵 조정

- 한 줄 요약: 피드백 이벤트 결과를 트리거 조건으로 로드맵 우선순위를 데이터 기반 조정
- 원래 형태: 인스타 콘텐츠 피벗에서 변형 — 콘텐츠 생산 중심에서 데이터 기반 의사결정으로 전환
- 사용자 가치: 사용자가 실제로 원하는 것을 먼저 만듦
- 실행 방법: 4개 트리거 조건 판단(3월 1주차) → 조건 충족 시 로드맵 조정
- 차별점: 1인 운영에서 "감"이 아닌 "데이터"로 우선순위 결정
- 판정: **승인**

## Decision Tree 결과

```
Q1. 사용자 가치가 명확한가? (파일럿 데이터/피드백 근거)
 ├─ NO → 기각
 └─ YES ↓

Q2. 1인 개발로 3개월 내 MVP 가능한가?
 ├─ NO → 스코프 축소 후 재평가
 └─ YES ↓

Q3. 가톨릭 전용 도구로서 차별점이 있는가?
 ├─ NO → 피벗 필요
 └─ YES ↓

Q4. 수익 모델/성장에 기여하는가?
 ├─ NO → 조건부 (성장 기여 보완 필요)
 └─ YES → 승인
```

| 전략 | Q1 사용자 가치 | Q2 3개월 내 MVP | Q3 차별점 | Q4 성장 기여 | 판정 |
|------|--------------|----------------|----------|------------|------|
| 전략 A: 계정 모델 전환 | YES (신당동/금호동 구조 혼란 실증) | YES (SDD 3월, 구현 4월) | YES (본당/모임 합류 구조는 가톨릭 특화) | YES (구조적 이탈 제거 → 리텐션) | **승인** |
| 전략 B: 피드백 기반 조정 | YES (피드백 이벤트 데이터) | YES (트리거 판단만, 개발 아님) | N/A (의사결정 프레임) | YES (데이터 기반 우선순위) | **승인** |
| 전략 C: 파일럿 확산 | YES (길음동/창동/가재울 실증) | YES (통화/DM, 개발 불필요) | YES (직접 온보딩은 범용 앱 불가) | YES (WAU 직접 기여) | **승인** |
