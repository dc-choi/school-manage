## 마이그레이션 단계

### 0단계: Parish(교구) 사전 적재

- 한국 가톨릭 교구 전체 데이터 사전 입력 (16개 교구)
- 서울대교구, 수원교구, 인천교구, 의정부교구, 대전교구, 청주교구, 춘천교구, 원주교구, 대구대교구, 부산교구, 안동교구, 마산교구, 광주대교구, 전주교구, 제주교구, 군종교구
- 마이그레이션 SQL에 INSERT로 포함 (마스터 데이터)

### 1단계: 조직 계정 마이그레이션 (20개)

본당 식별 가능한 Account → Church + Organization 자동 생성

1. 기존 Account마다 → Church 1개 + Organization 1개 자동 생성
   - Church.name: **본당-교구 매핑 테이블**에서 지정
   - Church.parishId: 매핑 테이블의 교구로 지정
   - Organization.name: displayName에서 조직명 추출
2. Account.organizationId = 생성된 Organization.id, Account.role = "ADMIN"
3. Group.accountId → Group.organizationId로 전환
4. Student.groupId → StudentGroup 레코드 생성 + Student.organizationId 설정
5. Attendance.groupId 유지 (변경 없음)

> 같은 본당에서 여러 Account → 같은 Church 하위에 별도 Organization 생성
> **실제 사례**: 구룡성당 주일학교(id:37) + 구룡성당 초등부(id:38) → 구룡 성당 Church 1개, Organization 2개

### 2단계: 미소속 계정 처리 (18개)

본당 식별 불가 + 제외 대상 + 확인 필요 계정

- Account.organizationId = null (조직 미소속 상태 유지)
- 재로그인 시 /join 화면으로 유도
  - 같은 본당의 조직이 이미 마이그레이션됨 → 해당 조직에 합류 요청 (관리자 승인)
  - 같은 본당의 조직이 없음 → 본당 + 조직 신규 생성 (본인이 admin)

> **황보나** (22학생, 32출석): 본당 불명으로 데이터가 붕뜸. /join 후 조직 생성하면 기존 데이터 재연결 필요 (수동 처리).

### 3단계: 공통

6. StudentSnapshot: organizationId 컬럼 추가 (기존 groupId 데이터 보존, 신규부터 organizationId 사용)
7. GroupSnapshot 유지 (변경 없음, 과거 데이터)
8. AccountSnapshot 초기 생성 (마이그레이션 시점 스냅샷, 조직 계정만)
