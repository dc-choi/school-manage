# PRD: 로그인 에러 메시지 통일 (사용자 열거 공격 대응)

> 상태: Draft | 작성일: 2026-04-21

## 배경/문제 요약

- 참고: `docs/specs/README.md` TARGET BUGFIX P2 "로그인 사용자 열거 공격"
- **문제**: 로그인 API가 "아이디 없음"과 "비밀번호 불일치"를 서로 다른 HTTP 상태 코드·메시지로 응답하여, 공격자가 유효한 아이디 목록을 추출할 수 있음 (OWASP Top 10 A07:2021, CWE-204)
- **현재 상태**:
    - 아이디 없음 → `TRPCError(NOT_FOUND, '존재하지 않는 아이디입니다.')` (HTTP 404)
    - 비밀번호 불일치 → `TRPCError(UNAUTHORIZED, '비밀번호가 일치하지 않습니다.')` (HTTP 401)
    - 두 응답이 구분 가능 → 공격자가 아이디 목록 dictionary attack 수행 가능
- **목표 상태**: 두 케이스 모두 동일한 에러 코드·메시지로 응답하여 유효한 아이디 식별 불가

## 목표/성공 기준

- **목표**: 외부에서 관찰 가능한 로그인 실패 응답을 단일화하여 사용자 열거 공격을 차단한다
- **성공 지표**:
    1. 두 케이스 응답의 HTTP 상태 코드·에러 코드·메시지 본문이 모두 동일
    2. 통합 테스트에서 두 케이스 모두 동일 기대 응답 검증 통과
- **측정 기간**: 배포 후 즉시 검증

## 사용자/대상

- **주요 사용자**: 출석부 앱 전체 사용자 (로그인 시도자)
- **사용 맥락**: 로그인 페이지에서 아이디·비밀번호 입력 후 제출. 실패 시 표시되는 에러 메시지를 보고 재시도

## 범위

### 포함

- `auth.login` tRPC procedure의 에러 응답 통일 (`login.usecase.ts`)
- 관련 통합 테스트 수정 (`apps/api/test/integration/auth.test.ts`)
- 프론트엔드 에러 메시지 표시 검증 (백엔드 메시지 그대로 표시하는 구조이므로 자동 반영 예상)

### 제외

- 회원가입/비밀번호 재설정 등 다른 인증 엔드포인트의 에러 메시지 통일 (별도 검토 필요)
- Rate limit 강화 (이미 구현됨, 본 PRD 범위 외)
- 타이밍 공격 대응 (현재도 bcrypt 비교를 항상 수행하므로 추가 조치 불필요)
- 로그 레벨 분리 (서버 내부 로그는 기존대로 구분 유지 가능)

## 사용자 시나리오

1. **정상 사용자 오타**: 사용자가 아이디 또는 비밀번호를 잘못 입력하면 "아이디 또는 비밀번호가 올바르지 않습니다." 통합 메시지를 본다. 사용자는 자신의 실수를 인지하고 재입력한다.
2. **공격자 시도**: 공격자가 무작위 아이디로 로그인을 시도해도, 응답이 아이디 존재 여부에 대한 힌트를 제공하지 않아 dictionary attack이 무의미해진다.

## 요구사항

### 필수 (Must)

- [ ] "아이디 없음"과 "비밀번호 불일치" 응답을 동일한 HTTP 401·에러 코드·메시지로 통일
- [ ] 통일 메시지: `'아이디 또는 비밀번호가 올바르지 않습니다.'`
- [ ] 에러 코드: `UNAUTHORIZED` (인증 실패 의미에 부합)
- [ ] 통합 테스트 2건 수정 (NOT_FOUND 기대 → UNAUTHORIZED 기대)
- [ ] 기존에 통과하던 성공 시나리오(정상 로그인) 테스트 영향 없음

### 선택 (Should)

- [ ] 프론트엔드에서 통일 메시지가 정상 표시되는지 수동 확인

### 제외 (Out)

- 회원가입 중복 검사 에러 메시지 (다른 위협 모델)
- 비밀번호 재설정 플로우 에러 메시지
- 로그인 시도 횟수 제한 추가 구현
- 서버 내부 로그 통일 (공격 탐지/디버깅 목적의 구분 로그는 유지 가능)

## 제약/가정/리스크/의존성

- **제약**:
    - 프론트엔드 UX 변경 최소화 (백엔드 메시지를 그대로 표시하는 구조 유지)
    - 기존 tRPC 에러 처리 패턴(`TRPCError` throw) 준수
- **가정**:
    - 프론트엔드 `LoginPage.tsx`가 백엔드 에러 메시지를 `extractErrorMessage()`로 그대로 표시하므로, 백엔드 변경만으로 프론트 메시지도 자동 반영됨
    - 사용자가 아이디 자체의 오타를 직접 식별해야 할 UX 요구는 없음 (비밀번호 관리자·복원 플로우로 대응 가능)
- **리스크**:
    - **리스크 1**: 사용자가 아이디 오타와 비밀번호 오타를 구분하지 못해 혼란 → 대응: 통합 메시지가 양쪽 가능성을 모두 안내하므로 사용자는 두 필드 모두 확인하게 됨. 기존 서비스들(Google, GitHub 등) 표준 관행
    - **리스크 2**: 기존 테스트가 NOT_FOUND 상태 코드를 기대하는 외부 코드가 있다면 깨짐 → 대응: 코드베이스 내 `auth.login` 호출부 grep 확인. tRPC 클라이언트는 인증 실패를 구분 처리하지 않음
- **내부 의존성**:
    - `login.usecase.ts` (에러 throw 지점)
    - `auth.test.ts` (통합 테스트)
- **외부 의존성**: 없음
